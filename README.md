## 提问prompt

如何向 AI 提问以获得高质量的知识学习文档。

### 核心原则

**给 AI 足够的约束，而不是足够的自由。** 越模糊的提问，AI 越容易给你"正确但无用"的泛泛之谈。你需要限定输出的**结构、深度和风格**。

### 万能提问模板

```
请为我生成一篇关于 [知识点] 的深度学习文档。

要求：
1. 目标读者：[你的水平，如"有 X 年经验的后端开发"]
2. 文档结构：
   - 一句话说清楚它是什么
   - 它解决什么问题（没有它会怎样）
   - 核心原理（用类比或图示辅助理解）
   - 关键概念逐一拆解
   - 实际代码示例（用 [语言]，场景要贴近真实业务）
   - 常见误区和踩坑点
   - 和相关技术的对比（如 [技术A] vs [技术B]）
   - 一张知识脑图总结
3. 风格：简洁直接，少用术语堆砌，多用"为什么"来串联逻辑
4. 长度：[大概多少字/多详细]
```

### 场景变体

#### 1. 从零理解一个概念（入门向）

```
我对 [概念] 完全不了解。请用"给聪明的外行人讲明白"的方式写一篇文档。
- 先用一个生活中的类比解释核心思想
- 再逐步引入技术细节
- 每引入一个新术语都要立刻解释
- 最后给出一个最小可运行的 demo
```

#### 2. 深挖底层原理（进阶向）

```
我已经会用 [技术/框架]，但想理解它底层是怎么工作的。请写一篇文档：
- 从源码/协议层面解释 [具体机制]
- 画出关键流程的执行时序
- 解释设计者为什么这样设计（trade-off 是什么）
- 和其他实现方案对比优劣
```

#### 3. 面试准备向

```
请帮我准备 [知识点] 的面试内容：
- 先给出这个知识点的完整知识体系（脑图形式）
- 列出高频面试问题（按难度分级）
- 每个问题给出"及格回答"和"加分回答"
- 标注哪些是容易被追问的点，以及追问方向
```

#### 4. 横向对比（选型/决策向）

```
请对比 [A] 和 [B]（可选：和 [C]）：
- 用表格对比核心维度（性能/易用性/生态/适用场景）
- 各自的典型使用场景
- 什么时候选 A，什么时候选 B
- 迁移成本和学习曲线
- 给出你的推荐及理由
```

### 提升质量的关键技巧

| 技巧 | 示例 |
|---|---|
| **声明你的背景** | "我有 3 年 Go 开发经验，熟悉 Kafka" → AI 会跳过基础废话 |
| **指定输出格式** | "用 Markdown，包含代码块和表格" → 结构更清晰 |
| **要求类比** | "用餐厅点餐的类比解释" → 抽象概念更好理解 |
| **要求反面案例** | "给出错误用法和正确用法的对比" → 印象更深刻 |
| **限定篇幅** | "3000字左右" → 避免又臭又长 |
| **追问而不是重问** | 第一轮生成后，针对薄弱环节追问，比如"第三部分的 XX 没讲清楚，展开说说" |

### 实际示例

学习 **Go 的 Context**：

```
请为我写一篇关于 Go Context 的深度学习文档。

背景：我有 3 年 Go 开发经验，日常写微服务，用过 context 但理解不深。

要求：
1. 一句话说清 context 的本质
2. 用一个实际场景（比如 HTTP 请求链路）说明没有 context 会出什么问题
3. 源码级别解释 context 的 4 种类型的实现原理
4. context 在 goroutine 树中的传播机制，画出传播示意图
5. 最佳实践和常见反模式（至少 5 个）
6. 面试高频问题和参考答案
7. 用 Markdown 格式，代码用 Go，3000 字左右
```

> **总结**：差的提问 → "请介绍一下 X"；好的提问 → 告诉 AI **你是谁、你要什么结构、要什么深度、要什么风格**。你给的约束越精确，AI 的输出越有用。
