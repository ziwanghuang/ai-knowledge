# 🛠️ 工程实践 — AI 应用落地

## 概述

AI 工程实践关注如何将 AI 技术从实验室原型转化为生产级应用。涵盖系统架构设计、性能优化、可观测性建设、成本控制、安全合规等方面，是 AI 应用真正产生业务价值的关键环节。

## 🏭 Demo vs 生产：AI 工程化落地的核心矛盾

> **核心论点**：写一个能跑的 Demo 只需要 1 天，做一个生产可用的 Agent 系统需要 3-6 个月。差距不在模型能力，而在**确定性、可控性、可观测性、可运维性**四个维度。

| 维度 | Demo | 生产 |
|------|------|------|
| 输入 | 精心构造的测试用例 | 任意用户的任意输入 |
| 输出 | "看起来对"就行 | 必须可审计、可溯源、结构化 |
| 错误 | 报错了重跑 | 不能报错，必须优雅降级 |
| 延迟 | 30 秒也能接受 | P99 < 3 秒 |
| 并发 | 1 个人测试 | 数百/数千并发用户 |
| 成本 | 无所谓 | 每个请求都要算账 |
| 安全 | 不考虑 | Prompt 注入、越权、PII 泄露 |
| 数据 | 几十篇测试文档 | 数万到数百万篇持续更新的文档 |
| 监控 | print 看日志 | 全链路 Tracing + 告警 |
| 评测 | "我觉得回答不错" | 自动化评测、回归门禁 |
| 可用性 | 能跑就行 | 99.9% SLA |

## 核心知识体系

### 1. 系统架构设计

- **LLM 应用架构模式**：
  - 单体式：简单直接，适合 MVP
  - 微服务式：模块解耦，独立扩展
  - 事件驱动式：异步处理，高吞吐
- **API 网关设计**：统一入口、负载均衡、限流熔断
- **缓存策略**：语义缓存（Semantic Cache）、精确缓存、多级缓存
- **异步处理**：消息队列、流式响应、长任务管理

### 2. LLM 输出不确定性控制 ⭐

> AI 工程化的"原罪"：传统软件 `f(x) = y` 是确定性的，LLM 天然是概率性的。

- **不确定性的三个层面**：
  | 层面 | 表现 | 影响 |
  |------|------|------|
  | 格式不确定性 | 要求返回 JSON 有时返回 Markdown | 下游解析崩溃 |
  | 内容不确定性 | 同一问题不同回答的信息量、准确度不同 | 用户体验不一致 |
  | 行为不确定性 | 同一场景 Agent 有时调工具 A 有时调工具 B | 流程不可控 |

- **格式确定性方案**：
  - 强制 JSON Schema 约束（OpenAI Structured Outputs、Anthropic Tool Use）
  - Pydantic 验证层：所有 LLM 输出经过 Schema 校验，不合规则重试
  - Instructor 库：自动处理 LLM 输出 → 结构化对象映射，内置重试
- **内容确定性方案**：
  - Temperature 设为 0（或极低值）
  - Prompt 模板化（Jinja2 管理变量和条件逻辑）
  - Few-shot 锚定：提供标准回答样例约束输出风格
  - Self-Consistency：多次采样取一致性最高的结果
- **行为确定性方案**：
  - 显式决策树：关键业务流程用代码逻辑控制，不依赖 LLM 自由推理
  - LangGraph 状态机：Agent 行为建模为有限状态机

> 🔑 **核心原则**：能用代码控制的就不要交给 LLM 决策。LLM 只负责"理解"和"生成"，流程控制交给确定性代码。

### 3. 性能与延迟优化 ⭐

**典型 RAG Agent 延迟拆解**：

| 环节 | 耗时 |
|------|------|
| Query 改写（LLM 调用） | ~800ms |
| 向量检索 | ~50ms |
| BM25 检索 | ~30ms |
| Cross-Encoder 重排 | ~200ms |
| LLM 生成（首 Token） | ~1500ms |
| LLM 生成（完整响应） | ~3000ms |
| 工具调用 | 500ms-5000ms |
| **总计** | **~6-10 秒** |

| 优化方向 | 技术手段 |
|---------|----------|
| 推理加速 | 模型量化、KV Cache、批处理、推测解码 |
| 网络优化 | 流式传输（SSE/WebSocket）、连接池、CDN |
| 缓存优化 | 语义缓存、Prompt 缓存、结果缓存 |
| 并发处理 | 异步 I/O、协程、线程池 |
| 成本优化 | 模型路由、Token 压缩、小模型分流 |

- **延迟优化专项策略**：
  | 策略 | 效果 | 实现方式 |
  |------|------|---------|
  | 流式输出（Streaming） | 用户 1-2 秒看到第一个字 | SSE / WebSocket 流式推送 |
  | 检索并行化 | Dense + BM25 并行执行 | asyncio.gather |
  | 语义缓存 | 相似问题直接返回缓存 | GPTCache / 自建语义缓存 |
  | 模型路由 | 简单问题用小模型 | 分类器判断问题复杂度 |
  | 预检索 | 用户打字时就开始检索 | 前端防抖 + 预检索 API |
  | 异步工具调用 | 工具调用与回答生成并行 | 先回答"正在处理"，完成后追加 |
  | Prompt 精简 | 减少不必要的上下文 | 按需加载 Few-shot 和指令 |
  | 推理优化 | 减少推理开销 | vLLM + PagedAttention + 量化 |

### 10. 高可用与容灾 ⭐

- **可用性目标**：99.9%（月停机 ≤ 43 分钟）、99.99%（月停机 ≤ 4.3 分钟）
- **故障点识别**：LLM API 不可用、向量数据库故障、工具服务故障、网络问题、限流过载
- **分级降级策略**：
  | 降级级别 | 触发条件 | 降级方案 |
  |---------|---------|----------|
  | Level 1 | LLM 超时/限流 | 切换备用 LLM（GPT-4 → Claude → 开源模型） |
  | Level 2 | 检索超时 | 跳过重排，直接用向量检索 Top-5 |
  | Level 3 | 向量数据库不可用 | 降级为 BM25 / 预编译 FAQ 缓存 |
  | Level 4 | 所有 AI 服务不可用 | 规则引擎匹配预设答案 / 转人工 |
- **LLM 多供应商策略**：
  - 多供应商优先级列表（OpenAI → Anthropic → 本地开源模型）
  - 熔断器模式（Circuit Breaker）：供应商连续失败自动熔断跳过
  - 每次调用设超时，超时后自动切换下一供应商
- **关键设计原则**：每级降级都有监控告警、降级自动触发无需人工、降级后尽快恢复正常路径

### 11. 版本管理与发布 ⭐

- **七轨版本耦合**：一次正确的 AI 系统发布需同时绑定：
  - Code（代码版本）
  - Model（模型版本）
  - Embedding（Embedding 模型版本）
  - Index（索引版本）
  - Prompt（Prompt 版本）
  - Eval Set（评测集版本）
  - Config（配置版本）
- **发布流水线**：构建 → 测试（单元+离线评测+安全）→ 灰度（5%→20%→50%→100%）→ 全量 → 监控观察
- **回滚策略**：
  | 回滚类型 | 时间 | 方式 |
  |---------|------|------|
  | Prompt 回滚 | 秒级 | 切换 Prompt 版本配置 |
  | 配置回滚 | 秒级 | Feature Flag / 配置中心 |
  | 代码回滚 | 分钟级 | K8s Rolling Update |
  | 索引回滚 | 分钟~小时级 | 切换索引别名指向旧版本 |
  | 模型回滚 | 分钟级 | 切换模型 endpoint |

### 12. 多语言与国际化（i18n）工程化 ⭐

> 当 Agent 系统需要服务多语言用户时，工程复杂度远超"翻译一下 Prompt"。

- **多语言 Prompt 管理**：
  | 挑战 | 说明 |
  |------|------|
  | 效果差异 | 英文 Prompt 效果通常优于中文，同一逻辑不同语言表现不同 |
  | 版本同步 | 中文版改了，英文版忘了同步 |
  | 评测独立 | 每种语言都需要独立的评测集和基线 |
  | 模型适配 | 不同模型对不同语言的支持能力差异大 |

- **跨语言 RAG**：
  - 用户用中文提问，但文档是英文的 → 需要跨语言检索能力
  - 方案 1：Query 翻译（用户中文 → 翻译为英文 → 检索英文文档）
  - 方案 2：多语言 Embedding（用多语言模型统一编码，如 BGE-M3、Cohere Multilingual）
  - 方案 3：双语索引（同一文档同时索引中英文版本）

- **多语言 Embedding 选型**：
  | 模型 | 多语言支持 | 特点 |
  |------|-----------|------|
  | BGE-M3 | 100+ 语言 | 开源，支持 Dense + Sparse + ColBERT |
  | Cohere Embed v3 | 100+ 语言 | 商用，内置压缩 |
  | E5-Mistral | 多语言 | 指令式 Embedding |
  | 语言专用模型 | 单语言 | 特定语言效果最好但不通用 |

- **输出语言控制**：确保 Agent 用用户的语言回答，不混杂其他语言（Prompt 中显式约束 + 输出语言检测）
- **文化适配**：不同地区的表达习惯、敏感话题、合规要求不同（如日期格式、货币单位、隐私法规）

### 13. 模型迁移与供应商锁定防护 ⭐

> 今天用 GPT-4，明天可能因为成本/合规/效果需要换成 Claude 或国产模型。如果架构没有做好抽象，迁移成本巨大。

- **模型 API 差异**：
  | 差异点 | OpenAI | Anthropic | 国产模型 |
  |--------|--------|-----------|----------|
  | 消息格式 | messages 数组 | messages 数组（略有不同） | 各家不同 |
  | Function Calling | 原生支持 | Tool Use | 部分支持 |
  | 流式输出 | SSE | SSE | SSE/WebSocket |
  | 结构化输出 | Structured Outputs | 部分支持 | 有限支持 |
  | 多模态 | 支持 | 支持 | 部分支持 |

- **Prompt 迁移难题**：
  - 在 GPT-4 上调好的 Prompt 换到 Claude 效果可能完全不同
  - 不同模型对指令的理解方式、遵循程度不同
  - Few-shot 示例的最优数量和格式因模型而异
  - **解决**：每个模型维护独立的 Prompt 版本 + 独立评测基线

- **抽象层设计**：
  | 方案 | 特点 | 适用场景 |
  |------|------|----------|
  | LiteLLM | 统一 API 代理，支持 100+ 模型 | 快速接入多供应商 |
  | OpenRouter | 云端模型路由服务 | 不想自建代理层 |
  | 自建抽象层 | 完全可控，可定制 | 有特殊需求的企业 |

- **迁移检查清单**：
  - [ ] API 格式适配完成
  - [ ] Prompt 针对新模型调优
  - [ ] Function Calling 兼容性验证
  - [ ] 评测基线在新模型上重新建立
  - [ ] 成本模型重新计算（不同供应商计费方式不同）
  - [ ] 延迟基线重新测量
  - [ ] 安全测试（不同模型的安全边界不同）

### 14. Agent 与遗留系统集成 ⭐

> Agent 不是孤立运行的，它需要与企业现有的各种系统对接。这些系统往往老旧、文档缺失、接口不标准。

- **遗留 API 适配**：
  | 挑战 | 解决方案 |
  |------|----------|
  | SOAP/XML 接口 | 适配层转换为 RESTful JSON |
  | 无文档的内部 API | 抓包分析 + 逆向文档化 + 封装为标准工具 |
  | 需要 VPN 的系统 | 部署网关代理在内网，Agent 通过网关访问 |
  | 页面操作型系统（无 API） | RPA + Playwright 自动化操作 |

- **数据格式转换**：
  - Agent 输出的结构化数据与遗留系统的数据格式映射
  - 字段名映射、数据类型转换、编码转换
  - 建立统一的数据转换层（Adapter Pattern）

- **事务一致性**：
  - Agent 操作跨多个系统时的分布式事务问题
  - Saga 模式：每步操作绑定补偿操作，失败时逆序补偿
  - 最终一致性：允许短暂不一致，通过异步对账修复

- **限流与配额**：
  - 遗留系统的 QPS 限制远低于 Agent 的调用频率
  - 解决：请求队列 + 令牌桶限流 + 批量合并请求

- **认证与授权统一管理**：
  - OAuth、SAML、API Key、Basic Auth 等多种认证方式
  - 建立统一的凭证管理服务（Vault / Secret Manager）
  - 凭证自动轮转、过期告警

- **数据同步延迟**：
  - Agent 写入系统 A 后，系统 B 还没同步到最新数据
  - 解决：写后读一致性检查 + 重试机制 + 最终一致性容忍

### 15. 容量规划与弹性伸缩 ⭐

> Agent 系统的资源消耗模式与传统 Web 服务不同：Token 消耗不可预测、LLM 调用延迟高、GPU 资源昂贵。

- **流量预估模型**：
  | 指标 | 预估方法 |
  |------|----------|
  | 日请求量 | 用户数 × 人均会话数 × 平均轮次 |
  | Token 消耗 | 平均请求 Token × 请求量（注意输入输出分别计费） |
  | 存储增长 | 文档增量 × Embedding 维度 × 4 bytes |
  | GPU 利用率 | 并发请求数 × 单请求推理时间 / GPU 数量 |

- **突发流量应对**：
  - 营销活动、热点事件导致的流量突增（可能 10-100 倍）
  - 预热机制：提前扩容 + 缓存预热
  - 排队机制：超过容量时排队等待，而非直接拒绝
  - 降级策略：流量过大时自动降级（用小模型、跳过重排等）

- **GPU 资源调度**（自部署场景）：
  - 多模型共享 GPU：不同模型按时间片或请求量共享
  - 动态批处理：积攒请求批量推理，提高 GPU 利用率
  - 弹性 GPU 集群：基于队列深度自动扩缩 GPU 节点

- **自动扩缩容策略**：
  | 触发指标 | 扩容阈值 | 缩容阈值 |
  |---------|---------|----------|
  | 请求队列深度 | > 100 | < 10 |
  | P99 延迟 | > 5s | < 2s |
  | GPU 利用率 | > 80% | < 30% |
  | Token 消耗速率 | 超过预算 120% | 低于预算 50% |

- **成本与性能的 SLO 平衡**：
  - 在预算约束下设定合理的 SLO（如：P99 < 5s，可用性 > 99.9%）
  - 成本超限时自动触发降级（模型路由到更便宜的模型）
  - 定期 Review 容量计划，根据实际数据调整

### 16. 组织与团队挑战 ⭐

> AI Agent 工程化不只是技术问题，组织和团队层面的挑战同样严重影响落地效果。

- **AI 工程师技能缺口**：
  - 既懂 ML/NLP 又懂后端工程的人才稀缺
  - 传统后端工程师需要学习 Prompt Engineering、RAG、评测方法论
  - ML 工程师需要学习生产级系统设计、可观测性、CI/CD

- **Prompt 工程的协作模式**：
  | 角色 | 职责 |
  |------|------|
  | 产品经理 | 定义业务场景、验收标准、用户体验要求 |
  | 领域专家 | 提供领域知识、审核回答质量、标注评测集 |
  | Prompt 工程师 | 设计和优化 Prompt、管理 Prompt 版本 |
  | 后端工程师 | 系统架构、工具开发、性能优化 |
  | 测试工程师 | 评测体系建设、Bad Case 管理、安全测试 |

- **AI 项目估期困难**：
  - LLM 的不确定性导致传统项目管理方法失效
  - Prompt 调优可能需要 1 天也可能需要 2 周，无法精确预估
  - 建议：采用迭代式开发，每个迭代设定明确的评测指标目标

- **期望管理**：
  - 业务方对 AI 能力的过高期望（"AI 应该什么都能做"）
  - 或过低期望（"AI 不靠谱，不敢用"）
  - 建议：用评测数据说话，明确 AI 的能力边界和适用场景

- **知识管理与经验沉淀**：
  - Prompt 调优经验、Bad Case 处理经验如何在团队中传承
  - 建立 Prompt Cookbook、Bad Case 知识库、最佳实践文档
  - 定期 Review 会议：分享调优经验、讨论疑难 Bad Case

- **On-Call 与故障响应**：
  - AI 系统的故障排查需要不同于传统系统的技能
  - 需要能看懂 Trace、理解 Prompt 逻辑、判断是模型问题还是数据问题
  - 建立 AI 系统专用的 Runbook 和故障排查手册

### 17. 工程化成熟度路线图 ⭐

| 阶段 | 时间 | 关键里程碑 |
|------|------|----------|
| Phase 0: Demo | 1-2 周 | 单脚本 RAG/Agent，证明想法可行 |
| Phase 1: MVP | 2-4 周 | API 服务化、流式输出、基础评测集（50+ 用例） |
| Phase 2: 可上线 | 1-2 月 | 结构化输出验证、工具安全层、安全审核、语义缓存、灰度发布 |
| Phase 3: 生产级 | 2-3 月 | 全链路 Tracing、多模型容灾、Bad Case 闭环、A/B 测试、HITL |
| Phase 4: 持续优化 | 持续 | 自动化 Bad Case 发现、在线学习、全链路血缘追踪、合规审计 |

- **成熟度检查清单维度**：可靠性、安全性、可观测性、评测、Prompt 管理、工具安全、成本、上下文管理、版本管理、数据管线

### 13. 编程与技术栈基础

- **编程语言**：Python（Agent 开发主力）、TypeScript/JavaScript（前端交互、MCP）、Go/Rust（高性能后端）
- **异步编程**：asyncio、并发控制
- **数据库**：PostgreSQL、Redis、MongoDB
- **消息队列**：Kafka、RabbitMQ（Agent 异步任务）
- **大数据处理（Spark）**：PySpark、大规模文本预处理、数据管道设计
- **软件工程实践**：设计模式（观察者、策略、责任链）、微服务架构、事件驱动架构
