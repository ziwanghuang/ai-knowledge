# 🤖 AI Agent — 智能体架构

## 概述

AI Agent（智能体）是指能够感知环境、自主决策并执行行动以完成目标的 AI 系统。与传统的单轮问答不同，Agent 具备规划、记忆、工具调用和多步推理能力，是 LLM 应用的高级形态。

## 核心知识体系

### 1. Agent 基础概念

- **感知-决策-行动循环**：Agent 的基本工作模式
- **自主性层级**：从简单工具调用到完全自主决策的演进
- **Cognitive Architectures**：感知-推理-行动循环的认知架构

### 2. 经典 Agent 范式

| 范式 | 说明 |
|------|------|
| **ReAct** | 推理 + 行动交替，最经典的 Agent 范式 |
| **Plan-and-Execute** | 先规划后执行的两阶段模式 |
| **Reflexion** | 自我反思与迭代改进，从失败中学习 |
| **LATS** | Language Agent Tree Search，树搜索 + LLM，探索多条执行路径 |

### 3. 核心能力模块

#### 3.1 🧠 规划（Planning）
- **任务分解（Task Decomposition）**：将复杂任务拆分为可执行的子任务
- **子目标生成**：为每个子任务设定明确目标
- **层次化规划（HTN）**：多层级的任务规划结构
- **动态重规划**：根据执行结果实时调整计划

#### 3.2 🔧 工具使用（Tool Use）
- **Function Calling**：模型理解和选择工具的核心机制
- **工具定义与参数设计**：JSON Schema 设计、参数类型与验证、描述撰写技巧（直接影响模型选择准确率）
- **工具选择与路由**：从大量工具中智能选取最相关的
- **工具组合策略**：多工具链式调用与并行调用
- **自定义工具开发与工具库设计**
- **执行类型**：API 调用、代码执行（Code Interpreter）、浏览器操作、文件系统操作

#### 3.3 💾 记忆系统（Memory）
- **短期记忆（Working Memory）**：上下文窗口内的即时信息
- **长期记忆**：向量数据库存储、关系数据库、知识图谱
- **情景记忆（Episodic Memory）**：具体事件和经历的记忆
- **语义记忆（Semantic Memory）**：概念和知识的抽象记忆
- **记忆检索与更新策略**：如何高效检索和更新记忆
- **记忆压缩（Memory Compression）**：对长对话历史进行摘要压缩，节省上下文窗口
- **记忆遗忘机制**：基于时间衰减、重要性评分的遗忘策略

#### 3.4 🔄 自我反思（Self-Reflection）
- **执行结果评估**：判断任务是否成功完成
- **错误纠正**：识别并修复执行中的错误
- **策略调整**：根据反馈优化后续行动策略

### 4. Function Calling 与 API 编排

- **Function Call 原理**：模型如何理解和选择工具
- **各大模型 FC 能力对比**：OpenAI / Claude / Qwen / DeepSeek
- **Tool Schema 定义**：JSON Schema 规范设计
- **函数调用链路**：工具选择 → 参数提取 → 执行 → 结果解析 → 多轮调用
- **API 网关与聚合**
- **错误处理与重试策略**
- **幂等性设计**

#### 4.1 工具调用工程化——Agent 最危险的部分 ⭐

> Agent + 工具 = 有"手"的 AI，手伸错了地方代价极高。

- **参数幻觉**：LLM 编造不存在的参数值（如虚假订单号）
  - 解决：参数 JSON Schema 校验 + 业务规则校验 + 枚举约束 + 外部数据库确认
- **越权调用**：Agent 调用了当前用户无权使用的工具
  - 解决：基于用户角色的动态工具列表注入 + 每次调用鉴权 + 行级数据过滤
- **幂等性缺失**：用户重试或网络重传导致工具被重复调用（如创建两个工单）
  - 解决：幂等 Key（基于请求哈希）+ 状态检查前置 + Saga 补偿事务
- **调用链故障处理**：多步工具调用中某步失败的恢复策略
  - 解决：断点续传 + 补偿机制（每个工具绑定 Undo 操作）+ 超时降级（转人工）+ 上下文敏感数据清理
- **工具描述质量**：描述写得不好导致 LLM 选错工具或传错参数
  - 最佳实践：说清楚什么时候用/不用、参数给具体示例、明确返回值结构、有限选项用 enum
- **工具数量膨胀**：50+ 工具时 LLM 选择准确率下降
  - 解决：工具路由器（先用向量检索筛 Top-5 候选）+ 工具分组（按场景两级路由）+ 动态加载

### 5. Multi-Agent 系统

- **协作模式**：多个 Agent 分工合作完成复杂任务
- **辩论模式（Debate）**：Agent 之间相互质疑与验证，提升输出质量
- **层级模式（Manager-Worker）**：Supervisor Agent 协调管理子 Agent
- **去中心化协作**：无中心节点的对等协作
- **通信协议**：Agent 之间的消息传递与状态共享机制

#### 5.1 Multi-Agent 生产级挑战 ⭐

- **通信协议不统一**：Agent 间输出格式不匹配导致解析失败
  - 解决：定义标准化 Agent 间消息协议（Message Schema），统一输入输出格式
- **无限循环 / 死锁**：Agent 间相互依赖或无限乒乓
  - 解决：最大轮次限制 + DAG 编排（不允许循环依赖）+ 超时熔断
- **成本失控**：多 Agent 协作 Token 消耗是单 Agent 的 3-5 倍，大量花在"内部对话"上
  - 解决：能单 Agent 解决就不用多 Agent + 分级路由 + 共享上下文避免重复检索
- **适用场景判断**：
  | 场景 | 推荐 | 原因 |
  |------|------|------|
  | 简单问答/客服 | ❌ 单 Agent | 不需要多角色 |
  | 代码审查 | ✅ 多 Agent | 不同角色关注不同方面 |
  | 复杂研究报告 | ✅ 多 Agent | 检索→分析→生成→审核流水线 |
  | 辩论式决策 | ✅ 多 Agent | 多角度论证提高决策质量 |
  | 日常工具调用 | ❌ 单 Agent | 过度设计 |

### 6. 关键设计模式

- **Router Pattern**：根据输入路由到不同的处理流程
- **Supervisor Pattern**：中心化的任务分配与监控
- **Hierarchical Pattern**：多层级的 Agent 组织结构
- **Tool-augmented Pattern**：通过外部工具扩展 Agent 能力

### 7. 上下文窗口管理——"无限记忆"的幻觉 ⭐

- **生产环境上下文膨胀问题**：System Prompt(3K) + 工具定义(5K) + RAG 结果(4K) + 对话历史(8K) + 用户输入(200) = 20K+ tokens/请求
- **核心挑战**：上下文溢出、注意力稀释（Lost-in-the-Middle）、成本线性增长、延迟增加、多轮记忆不一致
- **分层上下文管理架构**：
  | 层级 | 内容 | 存储 | 特点 |
  |------|------|------|------|
  | Layer 1: 即时上下文 | System Prompt + 最近 N 轮 + 当前 RAG 结果 | In-Context | 低延迟、高成本 |
  | Layer 2: 会话记忆 | 对话摘要 + 关键信息提取 | Redis / 内存数据库 | 中等延迟 |
  | Layer 3: 长期记忆 | 历史对话向量索引 + 用户画像 | 向量数据库 + 关系数据库 | 需检索触发 |
- **具体技术方案**：滑动窗口、对话摘要压缩、关键信息结构化提取、语义检索记忆、分层缓存

### 8. 状态管理与流程编排 ⭐

- **生产级状态管理需求**：Agent 交互可能在任何步骤中断（用户离开、网络断开、服务重启），恢复后必须从断点继续
- **核心难题**：持久化（不能只存内存）、并发安全、超时处理、错误恢复、状态可观测
- **技术选型**：
  | 方案 | 适用场景 | 特点 |
  |------|---------|------|
  | LangGraph | Agent 工作流编排 | 图结构、条件分支、内置持久化 |
  | Temporal | 长时间运行的工作流 | 自动重试、断点续传、强一致性 |
  | Dagster | 数据管线编排 | 资产化编排、回填、可观测 |
  | 自建状态机 | 简单流程 | 轻量但需自行处理持久化 |

> 🔑 **核心原则**：Agent 的业务流程用工作流引擎编排（确定性），LLM 只在每个节点内做"理解"和"生成"（概率性）。

### 9. Human-in-the-Loop（HITL）人机协作工程化 ⭐

> Agent 不是要取代人，而是要和人高效协作。生产环境中，很多操作必须有人工参与才能执行。

- **审批节点设计**：
  | 操作类型 | 是否需要人工审批 | 原因 |
  |---------|----------------|------|
  | 查询类（查订单、查余额） | ❌ 不需要 | 只读操作，无副作用 |
  | 创建类（创建工单、发通知） | 🟡 视场景 | 低风险可自动，高风险需审批 |
  | 修改类（修改配置、更新数据） | ✅ 需要 | 有副作用，需确认 |
  | 删除类（删除账号、撤销权限） | ✅ 必须 | 不可逆操作 |
  | 资金类（退款、转账） | ✅ 必须 | 涉及资金安全 |
  | 外部通知类（发邮件、发短信） | ✅ 需要 | 发出后无法撤回 |

- **异步等待机制**：
  - Agent 发起审批后挂起当前工作流，状态持久化到数据库
  - 设置审批超时时间（如 30 分钟），超时自动升级或取消
  - 支持审批人通过 Web/IM/邮件 等渠道完成审批
  - 审批完成后通过 Webhook/消息队列 恢复 Agent 流程

- **审批工作流**：
  - 审批队列管理：优先级排序、SLA 监控、超时升级
  - 多级审批：普通操作 → 一级审批，高危操作 → 二级审批
  - 审批记录：who/what/when/why 全记录，支持审计追溯

- **人机交接协议**：
  - Agent → 人工：传递上下文摘要（用户问题、已完成步骤、待处理事项、相关数据）
  - 人工 → Agent：处理结果回传，Agent 从断点恢复自动流程
  - 混合模式：Agent 辅助人工（生成建议 + 人工确认）vs 人工辅助 Agent（Agent 主导 + 人工兜底）

- **HITL 设计原则**：
  - 默认安全：不确定是否需要审批时，宁可多审批
  - 最小打扰：只在真正需要时才请求人工介入
  - 上下文完整：人工介入时能看到完整的决策上下文
  - 可配置：审批规则可通过配置动态调整，无需改代码

### 10. 实时性与事件驱动 Agent ⭐

> 当前大多数 Agent 是请求-响应模式（用户问 → Agent 答），但生产中还有大量事件驱动场景。

- **事件触发 Agent**：
  ```
  监控告警触发 → Agent 自动诊断根因 → 执行修复操作 → 通知相关人员
  数据异常检测 → Agent 分析异常原因 → 生成报告 → 推送给负责人
  用户行为事件 → Agent 实时个性化推荐 → 推送消息
  ```

- **长时间运行 Agent**：
  - 持续监控数据流，发现异常主动行动（而非等用户提问）
  - 需要心跳机制、健康检查、自动重启
  - 状态持久化：长时间运行中的状态不能丢失

- **定时任务 Agent**：
  - 每天自动生成业务报告
  - 定期检查数据质量、知识库时效性
  - 周期性执行巡检任务

- **事件驱动工程化挑战**：
  | 挑战 | 解决方案 |
  |------|----------|
  | Webhook 集成 | 标准化事件接收接口，统一事件格式 |
  | 事件去重与幂等 | 基于事件 ID 去重 + 幂等处理 |
  | 背压控制（Backpressure） | 事件队列 + 消费速率限制 + 优先级调度 |
  | 事件风暴 | 聚合/合并相似事件，避免 Agent 被淹没 |
  | 顺序保证 | 分区有序消费（Kafka Partition） |

- **技术选型**：
  | 方案 | 适用场景 |
  |------|----------|
  | Kafka + Agent Consumer | 高吞吐事件流处理 |
  | Temporal + 定时触发 | 定时任务 + 长时间工作流 |
  | CloudEvents + Webhook | 标准化事件接收 |
  | Redis Streams | 轻量级事件队列 |

### 11. 端侧 Agent 与边缘部署 ⭐

> 不是所有 Agent 都跑在云端。端侧部署是隐私敏感场景和离线场景的重要趋势。

- **端侧小模型选型**：
  | 模型 | 参数量 | 适用场景 |
  |------|--------|----------|
  | Phi-3 Mini | 3.8B | 端侧推理，逻辑能力强 |
  | Gemma 2B | 2B | 超轻量，移动端 |
  | Qwen-1.5B | 1.5B | 中文能力好，端侧部署 |
  | TinyLlama | 1.1B | 极致轻量 |
  | MLC-LLM | 多种 | 端侧推理框架，支持多模型 |

- **端云协同架构**：
  ```
  用户请求 → 端侧分类器判断复杂度
    → 简单任务：端侧小模型直接处理（低延迟、零成本、保隐私）
    → 复杂任务：上传到云端大模型处理
    → 网络不可用：端侧降级处理 + 缓存结果，联网后同步
  ```

- **离线能力**：
  - 预缓存高频问答和常用知识
  - 端侧向量检索（FAISS Lite / SQLite-VSS）
  - 网络恢复后自动同步离线期间的交互数据

- **资源约束优化**：
  | 约束 | 优化策略 |
  |------|----------|
  | 内存限制 | 4-bit 量化（GGUF）、模型分片加载 |
  | 算力限制 | 投机解码、KV Cache 优化、批处理 |
  | 电量限制 | 动态调整推理精度、空闲时预计算 |
  | 存储限制 | 模型压缩、按需下载模型组件 |

- **模型更新**：OTA 增量更新机制、A/B 测试端侧模型版本、回滚策略
