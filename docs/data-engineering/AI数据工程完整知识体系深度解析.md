## 一、数据收集与获取

### 1.1 数据源全景图

AI 系统的数据来源可以按**获取难度**和**数据质量**两个维度分层：

```
                    高质量
                      │
   企业内部数据 ◄──────┤──────► 专业标注数据集
   (结构化/日志/文档)   │     (Alpaca/FLAN/Dolly)
                      │
   合作获取数据 ◄──────┤──────► 公开数据集
   (行业联盟/数据交换)  │     (Common Crawl/Wikipedia)
                      │
   用户生成数据 ◄──────┤──────► Web 爬虫数据
   (交互反馈/UGC)      │     (未经处理)
                      │
                    低质量
        低获取难度 ◄────┼────► 高获取难度
```

**四大数据来源详解：**

| 来源类型 | 代表 | 优点 | 缺点 | 典型规模 |
|---------|------|------|------|---------|
| 公开数据集 | Common Crawl、Wikipedia、BookCorpus、The Pile | 免费、规模大 | 质量参差不齐、可能有版权问题 | TB 级 |
| Web 爬虫采集 | 自建爬虫、Scrapy 集群 | 可定制、覆盖面广 | 合规风险、反爬对抗、清洗成本高 | 取决于目标 |
| 用户生成数据 | 产品日志、用户反馈、对话记录 | 真实场景、高相关性 | 需要产品先跑起来、隐私问题 | 取决于用户量 |
| 合作/购买 | 行业数据供应商、数据交换平台 | 高质量、合规 | 成本高、可能不独占 | 视合同而定 |

### 1.2 数据采集的合规性 ⚠️

这是**不可跳过的第一道关卡**。不合规的数据源可能让整个项目报废。

**三大隐私法规对比：**

| 法规 | 地域 | 核心要求 | 违规处罚 |
|------|------|---------|---------|
| **GDPR** (欧盟) | 欧盟+EEA | 明确同意、数据最小化、被遗忘权、DPO（数据保护官） | 最高 2000 万欧元或全球营收 4% |
| **CCPA/CPRA** (美国加州) | 加州居民 | 知情权、删除权、拒绝出售权 | 每次违规 $2,500-$7,500 |
| **个人信息保护法** (中国) | 中国境内 | 知情同意、最小必要、跨境传输限制 | 最高 5000 万元或年营收 5% |

**数据采集合规检查清单：**

```
□ 数据来源是否有合法授权/公开许可？
□ 是否涉及个人身份信息 (PII)？
□ robots.txt 和 ToS 是否允许爬取？
□ 训练用途是否属于"合理使用"（Fair Use）？
□ 跨境传输是否满足当地法规？
□ 是否有数据保留与删除策略？
□ 用户是否被告知数据用途？
```

> **📌 面试高频**: 被问到数据合规时，能说出 GDPR 的"合法基础六选一"（consent, contract, legal obligation, vital interests, public task, legitimate interests）是加分项。

**核心组件技术选型：**

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| 爬虫框架 | Scrapy + scrapy-redis | 分布式支持好，中间件生态丰富 |

| 分布式存储 | HDFS / S3 / MinIO | 原始数据落盘 |

#### 反爬虫对抗与合规边界

反爬虫是一场持续的攻防战，但**合规是不可逾越的底线**：

| 反爬手段 | 对抗方案 | 合规性 |
|---------|---------|--------|
| IP 限流 | 代理池 (Proxy Pool) | ✅ 合规 |
| User-Agent 检测 | UA 轮换 | ✅ 合规 |
| Cookie/Session 验证 | 会话管理 | ⚠️ 需查看 ToS |
| 验证码 | 打码平台/模型识别 | ⚠️ 灰色地带，部分场景违规 |
| 登录墙 | 自动登录 | ❌ 通常违反 ToS |
| 法律手段 (robots.txt, CFAA) | 尊重规则 | 必须遵守 |

> **原则**: 如果 robots.txt 禁止了，就别爬。如果 ToS 明确禁止爬取用于 AI 训练，就别爬。这不是技术问题，是法律问题。

#### 数据质量过滤流水线

原始爬取数据到可用训练数据，需要经过**多级过滤**：

```
原始数据 (100%)
    ├─ 语言检测 (fasttext-lid) ──────────── 过滤非目标语言 → ~70%
    ├─ 去重
    │   ├─ URL 级精确去重 ─────────────── 完全相同 URL → ~60%
    │   ├─ 文档级 MinHash 近似去重 ─────── 高度相似文档 → ~40%
    │   └─ 段落级去重 ─────────────────── 模板/样板文本 → ~35%
    ├─ 质量过滤
    │   ├─ 长度过滤 (太短/太长) ─────────── → ~30%
    │   ├─ 困惑度过滤 (PPL) ────────────── 非自然语言 → ~25%
    │   ├─ 质量分类器 ─────────────────── 低质量内容 → ~15%
    │   └─ 启发式规则 (特殊字符比例、重复率) ─ → ~12%
    ├─ 毒性检测
    │   ├─ 有害内容分类器 ─────────────── 暴力/仇恨/色情 → ~10%
    │   └─ PII 检测与脱敏 ────────────── 个人信息 → ~10%
    └─ 最终可用数据 (~10%)
```

**关键过滤技术解析：**

**MinHash 近似去重** — 核心思想是将文档转化为签名（sketch），然后通过签名相似度近似文档相似度：

```python
# MinHash 去重的核心流程（概念代码）
def minhash_dedup(documents, num_perm=128, threshold=0.8):
    """
    1. 将每个文档 tokenize 为 n-gram 集合
    2. 对每个集合计算 MinHash 签名（num_perm 个哈希函数）
    3. 使用 LSH (Locality-Sensitive Hashing) 将相似签名分到同一桶
    4. 同一桶内的文档两两比较 Jaccard 相似度
    5. 相似度 > threshold 的保留一份
    """
    # 实际生产中使用 datasketch 库
    from datasketch import MinHash, MinHashLSH
    
    lsh = MinHashLSH(threshold=threshold, num_perm=num_perm)
    for doc_id, doc in enumerate(documents):
        mh = MinHash(num_perm=num_perm)
        for ngram in get_ngrams(doc, n=5):
            mh.update(ngram.encode('utf8'))
        lsh.insert(str(doc_id), mh)
    
    # 查询每个文档的近似重复
    duplicates = set()
    for doc_id, doc in enumerate(documents):
        result = lsh.query(minhashes[doc_id])
        if len(result) > 1:
            duplicates.update(result[1:])  # 保留第一个
    
    return [d for i, d in enumerate(documents) if str(i) not in duplicates]
```

**困惑度过滤 (Perplexity Filtering)** — 使用一个预训练语言模型来评估文本的"自然程度"：

- **高困惑度 = 不自然** → 可能是乱码、机器翻译、关键词堆砌
- **极低困惑度 = 太模式化** → 可能是模板文本、重复内容
- 通常选择困惑度在 **中等范围** 的文本
- 常用工具：KenLM（轻量级 n-gram 语言模型，适合大规模过滤）

### 1.4 API 数据集成

结构化数据通过 API 获取，关键是**标准化和增量同步**：

```
外部 API ──→ API Gateway ──→ Schema 映射 ──→ 数据标准化 ──→ 存储
   │                             │                │
   │                        ┌────┴────┐      ┌───┴───┐
   │                        │ 字段映射  │      │ 类型转换│
   │                        │ 缺失填充  │      │ 编码统一│
   │                        │ 格式对齐  │      │ 时间标准│
   │                        └─────────┘      └───────┘
   │
   ├─ 全量拉取 (首次)
   ├─ 增量同步 (webhook / polling)
   └─ 变更追踪 (CDC)
```

**API 数据集成最佳实践：**
- **限流管理**: 使用令牌桶/漏桶算法，避免触发 API Rate Limit
- **幂等重试**: 网络失败时自动重试，确保数据不重复不丢失
- **Schema 演进**: API 可能变更字段，需要 Schema Registry 管理版本
- **监控告警**: 数据量异常波动时告警（可能是 API 变更或数据源故障）

## 二、数据标注

### 2.1 标注类型与任务设计

不同的 AI 任务需要不同类型的标注，标注设计直接决定了模型能力的上限：

| 标注任务 | 数据格式 | 用途 | 难度 | 成本/条 |
|---------|---------|------|------|---------|
| 文本分类 | `(text, label)` | 情感分析、意图识别 | 低 | ¥0.1-0.5 |
| 序列标注 (NER) | `(text, [(start, end, type)])` | 实体识别 | 中 | ¥0.5-2 |
| 关系标注 | `(text, [(entity1, relation, entity2)])` | 关系抽取、知识图谱 | 高 | ¥1-5 |
| 指令-回答对 | `(instruction, response)` | SFT 微调 | 高 | ¥2-10 |
| 偏好标注 | `(prompt, chosen, rejected)` | RLHF/DPO 对齐 | 很高 | ¥5-20 |
| 多模态标注 | `(image, text_description)` | VLM 训练 | 高 | ¥2-15 |

#### 偏好标注的特殊性

偏好标注是 RLHF/DPO 的核心数据来源，也是标注中**最难、最主观**的部分：

```
Prompt: "解释量子计算的基本原理"

Response A (chosen):
"量子计算利用量子力学原理（叠加态和纠缠）进行计算。
 传统比特只能是 0 或 1，量子比特可以同时是 0 和 1 的叠加态..."

Response B (rejected):
"量子计算是一种很先进的技术，它使用量子来计算东西。
 很多大公司都在研究量子计算..."

标注维度:
- 准确性: A > B (A 有具体技术细节)
- 有用性: A > B (A 直接回答了原理)
- 安全性: A = B (都无安全问题)
- 最终偏好: A > B
```

**偏好标注的核心挑战：**
1. **主观性高**: 不同标注员可能有不同偏好
2. **位置偏见**: 标注员倾向选择前面或更长的答案
3. **一致性差**: 同一标注员不同时间可能做出不同判断
4. **领域依赖**: 需要具备领域知识才能判断准确性

### 2.2 标注质量控制

标注质量是模型效果的上限。**垃圾标注 = 垃圾模型**，没有例外。

#### 质量控制体系

```
         ┌──────────────────────────────────┐
         │          标注质量控制体系          │
         └──────────────────┬───────────────┘
                            │
          ┌─────────────────┼─────────────────┐
          │                 │                 │
    ┌─────▼─────┐    ┌─────▼─────┐    ┌─────▼─────┐
    │ 事前预防   │    │ 事中监控   │    │ 事后审核   │
    └─────┬─────┘    └─────┬─────┘    └─────┬─────┘
          │                │                │
    ├─ 标注指南撰写     ├─ 实时一致性检查    ├─ 交叉验证
    ├─ 标注员培训      ├─ 金标准题抽查     ├─ 专家审核
    ├─ 试标注与校准    ├─ 标注速度监控     ├─ 统计分析
    └─ 标注界面优化    └─ 争议解决机制     └─ 反馈迭代
```

#### 一致性指标

**Cohen's Kappa (两人标注一致性)**:

```
        P(observed) - P(expected)
κ = ─────────────────────────────
        1 - P(expected)

- κ < 0.2: 几乎没有一致性 → 标注指南有严重问题
- 0.2 ≤ κ < 0.4: 一般一致性 → 需要培训和校准
- 0.4 ≤ κ < 0.6: 中等一致性 → 可接受但需改进
- 0.6 ≤ κ < 0.8: 较好一致性 → 大多数任务可接受
- κ ≥ 0.8: 几乎完美一致 → 理想状态
```

**Fleiss' Kappa (多人标注一致性)**: 扩展到多于两个标注员的场景，计算公式类似但考虑了所有标注员的一致性。

**实用建议**:
- 每个样本至少 **2-3 人标注**，取多数投票或加权平均
- 保留 5-10% 的**金标准题**（已知正确答案），混入标注任务中监控标注员质量
- 对低一致性样本进行**讨论会标注**（adjudication），达成共识后作为标准

### 2.3 标注平台选型

| 平台 | 类型 | 核心优势 | 适用场景 | 价格 |
|------|------|---------|---------|------|
| **Label Studio** | 开源自部署 | 灵活、可扩展、支持多模态 | 中小团队自主标注 | 免费（开源）|
| **Prodigy** | 商业桌面 | 主动学习集成、效率极高 | 数据科学家个人/小团队 | $390/年/人 |
| **Scale AI** | 托管服务 | 大规模标注团队、质量有保障 | 大公司、大规模标注需求 | 按量计费 |
| **Amazon SageMaker GT** | 云服务 | 与 AWS 深度集成、Mechanical Turk | AWS 生态用户 | 按量计费 |
| **Labelbox** | SaaS | 协作功能强、自动化标注 | 中大型团队 | 分层定价 |

**选型决策树**：

```
需要标注 → 预算多大？
  ├─ 几乎没预算 → Label Studio (开源) + 内部标注
  ├─ 中等预算 → Prodigy (效率高) 或 Label Studio Enterprise
  └─ 充足预算 → Scale AI (托管) 或 Labelbox (SaaS)
       ├─ 需要大规模标注 → Scale AI
       └─ 需要团队协作 → Labelbox
```

### 2.4 主动学习 (Active Learning) 降低标注成本

**核心思想**: 不是所有数据都值得标注。让模型"告诉"你哪些数据最需要人类标注。

```
                    ┌──────────────┐
                    │  未标注数据池   │
                    │  (大量、便宜)  │
                    └──────┬───────┘
                           │ 采样策略
                    ┌──────▼───────┐
                    │  选择最有价值   │
                    │  的样本进行标注  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  人工标注      │
                    │  (少量、精确)  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  更新模型      │◄────── 循环
                    └──────────────┘
```

**三大采样策略**：

| 策略 | 思想 | 数学表达 | 适用场景 |
|------|------|---------|---------|
| **不确定性采样** | 选模型最不确定的样本 | max H(y\|x) 或 min max P(y\|x) | 分类任务 |
| **多样性采样** | 选能最大化覆盖面的样本 | 基于聚类或核心集 (Coreset) | 数据分布不均匀 |
| **混合策略** | 同时考虑不确定性和多样性 | α·uncertainty + β·diversity | 大多数场景推荐 |

**效果数据**: 主动学习通常可以用 **20-30% 的标注量** 达到全量标注 **90%+ 的效果**。在标注成本高的场景（如医疗、法律），ROI 极高。

> **📌 面试高频**: "如何在有限预算下最大化标注效果？" → 主动学习 + LLM 辅助预标注 + 人工审核 的三段式流程。

## 三、合成数据

### 3.1 为什么合成数据变得如此重要

三个核心驱动力：

1. **真实数据获取成本高**: 标注一条高质量偏好数据可能需要 ¥10-20
2. **数据隐私限制**: 医疗、金融等领域真实数据不能随意使用
3. **长尾场景覆盖**: 真实数据中罕见场景的样本不足

**合成数据的黄金公式**:

```
模型效果 = f(数据质量, 数据多样性, 数据规模)

其中:
- 数据质量: 合成 > 低质量真实数据 (垃圾标注)
- 数据多样性: 合成数据可以系统性覆盖长尾
- 数据规模: 合成数据几乎无上限
```

### 3.2 LLM 生成合成数据的方法论

#### Self-Instruct

**论文**: Wang et al., 2023 — "Self-Instruct: Aligning Language Models with Self-Generated Instructions"

**核心流程**:

```
第一步: 准备种子任务池 (175 条人工编写的 instruction-response 对)
                    │
第二步: 随机采样种子任务作为 few-shot 示例
                    │
第三步: 让 LLM 生成新的指令 ──────────────┐
                    │                     │
第四步: 过滤低质量/重复指令                  │ 循环
                    │                     │
第五步: 让 LLM 为新指令生成回答              │
                    │                     │
第六步: 将合格的 (instruction, response)    │
       加入种子池 ──────────────────────────┘
```

**Self-Instruct 的关键 Prompt 设计**:

```
Come up with a series of tasks:

Task 1: {seed_task_1}
Task 2: {seed_task_2}
Task 3: {seed_task_3}

Task 4:
```

**优势**: 简单、通用、不需要额外标注
**劣势**: 生成的指令往往偏简单、缺乏多样性、容易"塌缩"到某些模式

#### Evol-Instruct (WizardLM)

**论文**: Xu et al., 2023 — "WizardLM: Empowering Large Language Models to Follow Complex Instructions"

**核心创新**: 不是从零生成，而是**渐进式进化**现有指令的复杂度。

```
原始指令 ───→ 进化操作 ───→ 更复杂的指令
  │               │
  │     ┌─────────┼─────────┐
  │     │         │         │
  │  深度进化    广度进化    其他
  │     │         │
  │  ├─ 增加约束  ├─ 增加步骤
  │  ├─ 加难条件  ├─ 替换主题
  │  ├─ 具体化   ├─ 组合任务
  │  └─ 增加推理  └─ 换场景
  │
  └── 评估 & 过滤 ───→ 保留合格的进化指令
```

**进化 Prompt 示例 (深度进化 - 增加约束)**:

```
I want you to act as a Prompt Rewriter.
Your objective is to rewrite a given prompt into a more complex version.

The rewriting should:
1. Add one more constraint/requirement to the original prompt
2. The added constraint should be reasonable and not too artificial
3. The rewritten prompt should be natural and not too verbose

Original prompt: Write a Python function that sorts a list.

Rewritten prompt: Write a Python function that sorts a list of 
dictionaries by a specified key, handling cases where the key might 
be missing in some dictionaries, and support both ascending and 
descending order.
```

**效果**: WizardLM 70B 在 MT-Bench 上接近 GPT-3.5-turbo，证明指令复杂度提升对模型能力影响巨大。

#### GLAN (Generalized Instruction Tuning)

**核心思想**: 不靠随机采样或进化，而是**系统性地枚举人类知识领域**，为每个领域生成指令数据。

```
第一步: 构建学科分类树
        人类知识
        ├─ 自然科学
        │   ├─ 物理
        │   │   ├─ 力学
        │   │   ├─ 电磁学
        │   │   └─ ...
        │   ├─ 化学
        │   └─ 生物
        ├─ 工程技术
        │   ├─ 计算机科学
        │   │   ├─ 算法
        │   │   ├─ 系统设计
        │   │   └─ ...
        │   └─ ...
        └─ 人文社科
            └─ ...

第二步: 为每个叶子节点生成多种类型的指令
        (问答、分析、创作、代码、推理、总结...)

第三步: 由 LLM 生成回答

第四步: 质量过滤
```

**优势**: 多样性有保障（覆盖完整知识图谱），适合训练通用型模型

### 3.3 合成数据的质量控制

合成数据的最大风险是**看起来很多但实际有效信息很少**。

#### 去重与去污染

```
合成数据集
    │
    ├─ 内部去重: 合成数据之间的重复
    │   ├─ 精确去重 (MD5/SHA256)
    │   └─ 语义去重 (Embedding 相似度 > 0.95 → 去掉)
    │
    ├─ 外部去重: 与测试集的重复（数据污染）
    │   ├─ 精确匹配: 检查测试集的 question/answer 是否在训练集中出现
    │   └─ 近似匹配: n-gram overlap > 阈值 → 标记
    │
    └─ 与原始种子的距离检查
        └─ 如果合成数据过于接近种子数据 → 多样性不足
```

#### 多样性评估

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| 词汇多样性 | Unique n-gram / Total n-gram | > 0.8 |
| 主题分布 | Topic Model 聚类后的分布均匀度 | 接近均匀 |
| 难度分布 | 按指令复杂度分级后的分布 | 梯度分布 |
| 格式多样性 | 统计不同回答格式的占比 | 多种格式 |
| Embedding 空间覆盖 | 嵌入后的空间分布（可视化） | 均匀分散 |

### 3.4 合成数据的陷阱

#### Model Collapse（模型崩塌）

**问题**: 用模型 A 的输出训练模型 B，再用 B 的输出训练 C... 经过多轮递归，模型输出的多样性会**急剧下降**，最终塌缩到少数模式。

```
原始分布: [正态分布，多样性高]
        ↓ 训练模型 1
第1代合成: [分布稍窄]
        ↓ 训练模型 2
第2代合成: [分布明显变窄]
        ↓ 训练模型 3
第3代合成: [几乎只有几种模式] ← 崩塌！
```

**论文**: Shumailov et al., 2024 — "AI models collapse when trained on recursively generated data" (Nature)

**应对策略**:
1. **始终混入真实数据**: 合成:真实 = 7:3 ~ 5:5
2. **控制递归深度**: 不要用合成数据训练出的模型再生成合成数据
3. **多样性监控**: 每批次检查指标，发现塌缩苗头立刻停止
4. **多模型交叉生成**: 用模型 A 生成的数据训练模型 B，而非自我循环

#### 同质化（Homogenization）

LLM 生成的文本天然具有某种"LLM 风格"（如过度使用"首先...其次...最后..."句式），大量合成数据会让微调模型也沾染这种风格。

**应对**:
- 使用多个不同 LLM 生成（GPT-4 + Claude + Gemini 混合）
- 设定不同的 temperature 和 system prompt
- 后处理阶段做风格多样化

### 3.5 合成 vs 真实数据的混合策略

**最佳实践是混合使用**，不要只依赖一种：

```
推荐比例 (根据场景调整):

通用指令微调:
  合成指令: 70%  (Self-Instruct + Evol-Instruct)
  真实指令: 30%  (用户真实查询)

偏好对齐 (RLHF/DPO):
  合成偏好: 40%  (Constitutional AI + 模型排序)
  真实偏好: 60%  (人工标注)  ← 偏好数据质量更关键

领域微调:
  合成领域数据: 50%  (基于种子数据扩写)
  真实领域数据: 50%  (领域文档/对话)
```

> **📌 面试高频**: "合成数据能完全替代真实数据吗？" → 不能。合成数据解决的是"量"和"多样性"，但"分布真实性"和"领域特异性"仍然需要真实数据。最优策略是混合使用。

## 四、数据飞轮

### 4.1 数据飞轮的核心理念

**数据飞轮 (Data Flywheel)** 是 AI 产品最强的护城河——不是技术壁垒，而是**数据壁垒**。

```
                    ┌─────────────────┐
                    │  更多用户使用     │
                    │  (用户增长)      │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
              ┌─────│  更好的产品体验   │─────┐
              │     │  (留存提升)      │     │
              │     └─────────────────┘     │
              │                             │
     ┌────────▼────────┐          ┌────────▼────────┐
     │  模型持续优化     │          │  收集用户数据     │
     │  (微调/RAG更新)  │          │  (交互/反馈/行为) │
     └────────┬────────┘          └────────┬────────┘
              │                             │
              │     ┌─────────────────┐     │
              └─────│  数据清洗标注     │─────┘
                    │  (构建训练集)    │
                    └─────────────────┘
```

**核心洞察**: 飞轮的每一圈都让下一圈更容易——用户越多 → 数据越多 → 模型越好 → 体验越好 → 用户更多。先发者一旦转起来，后来者很难追上。

**经典案例**:
- **Tesla FSD**: 每辆车都是数据采集器，累积数十亿英里真实驾驶数据
- **GitHub Copilot**: 用户使用 → 收集采纳率 → 优化建议质量 → 更多使用
- **ByteDance 推荐系统**: 用户行为 → 优化推荐 → 更多停留 → 更多行为数据

### 4.2 用户反馈数据收集

反馈数据是飞轮的"输入端"。收集方式分为**显式**和**隐式**：

#### 显式反馈

```
用户主动给出的评价信号:

👍 / 👎         → 二元偏好信号 (成本最低，信息量也最低)
⭐⭐⭐⭐⭐ 评分  → 多级评分 (更细粒度)
"这个回答不对，应该是..."  → 纠正性反馈 (最有价值！)
"请重新生成"    → 隐含不满 (介于显式和隐式之间)
```

**显式反馈的收集设计要点**:
1. **低摩擦**: 反馈操作越简单越好（单次点击优于文字输入）
2. **适时触发**: 在用户完成关键动作后触发（如复制了代码 → 询问是否有用）
3. **避免打扰**: 不要每次都弹窗，可以用抽样策略
4. **反馈闭环**: 告知用户反馈被采纳了（增强反馈意愿）

#### 隐式反馈

```
通过用户行为推断满意度:

行为信号              推断逻辑                     信号强度
──────────          ──────────                   ─────────
复制代码/文本        → 用户认为内容有用              强正向 ✅
停留时间长           → 用户在认真阅读              弱正向
点击"重新生成"       → 对当前回答不满              强负向 ❌
回答后立即离开        → 可能没解决问题              弱负向
追问相关问题         → 对话有效，继续深入           正向 ✅
切换到其他工具       → 当前工具没能满足需求          负向
会话持续时间         → 较长可能是好事也可能是卡住     模糊信号 ⚠️
```

**隐式反馈的价值**: 量大、无需用户配合，但**信号噪声比低**，需要组合多个信号综合判断。

#### 对话级 vs 回答级反馈

| 粒度 | 含义 | 适用场景 | 数据利用 |
|------|------|---------|---------|
| **回答级** | 对单条回答的评价 | SFT 数据、偏好对数据 | 直接构造训练样本 |
| **对话级** | 对整个对话的评价 | 对话策略优化 | 用于 RL 的 reward |
| **任务级** | 任务是否最终完成 | Agent 系统 | 长程 reward signal |

从用户反馈到模型优化，需要一套**自动化的数据管道**：

### 4.4 A/B 测试中的数据收集

A/B 测试是验证数据飞轮效果的**唯一可靠手段**：

```
用户流量
    │
    ├─ 50% ──→ 实验组 (新模型/新 Prompt)
    │              │
    │              ├─ 收集: 满意度、任务完成率、Token 消耗
    │              └─ 时长: 至少 1-2 周
    │
    └─ 50% ──→ 对照组 (原模型/原 Prompt)
                   │
                   ├─ 收集: 相同指标
                   └─ 确保: 相同时间段

统计检验:
    - 使用 t 检验 / Mann-Whitney U 检验
    - p < 0.05 且效应量 (Effect Size) 有实际意义 → 采纳
    - 注意 SRM (Sample Ratio Mismatch) 检查
```

**分流策略要点**:
- 按用户 ID Hash 分流，确保同一用户始终在同一组
- 避免"新奇效应"（novelty effect）：新功能初期数据偏高
- 考虑"学习效应"：用户适应新功能需要时间

### 4.5 冷启动问题

没有用户就没有数据，没有数据就没有好模型，没有好模型就吸引不来用户。**鸡生蛋问题**。

**冷启动策略金字塔**:

```
┌────────────────────────────────┐
│  第四阶段: 数据飞轮自动运转     │  用户 > 1000
│  真实反馈驱动持续改进           │
├────────────────────────────────┤
│  第三阶段: 种子用户反馈         │  用户 100-1000
│  邀请内测用户，收集真实场景数据  │
├────────────────────────────────┤
│  第二阶段: 合成数据 + 人工评估   │  用户 < 100
│  用 LLM 生成训练数据，人工校验   │
├────────────────────────────────┤
│  第一阶段: 公开数据 + 通用模型   │  用户 = 0
│  基于开源数据和通用模型先跑通    │
└────────────────────────────────┘
```

> **📌 面试高频**: "如何设计数据飞轮？" → 四阶段回答法：公开数据冷启 → 合成数据补充 → 种子用户反馈 → 全量飞轮运转。关键是**从 Day 1 就埋好反馈收集点**。

## 五、数据个性化

### 5.1 用户画像构建

个性化的基础是**了解用户**。用户画像从多维数据中构建：

```
用户画像 (User Profile)
    │
    ├─ 静态属性 (注册时获取)
    │   ├─ 角色/职位: 开发者、产品经理、学生
    │   ├─ 行业/领域: 金融、医疗、教育
    │   └─ 语言/地区: 中文、英文
    │
    ├─ 行为特征 (交互中提取)
    │   ├─ 提问风格: 简洁型 vs 详细型
    │   ├─ 知识水平: 入门 vs 高级 (根据提问复杂度推断)
    │   ├─ 偏好格式: 代码多 vs 文字多 vs 图表多
    │   ├─ 活跃时段: 工作日 vs 周末、上午 vs 下午
    │   └─ 常用功能: 搜索、代码生成、文档写作
    │
    └─ 反馈历史 (累积的评价)
        ├─ 满意的回答有什么特征？
        ├─ 不满意的回答有什么特征？
        └─ 哪些话题回答质量高/低？
```

**用户画像构建的技术方案**:

```python
# 用户画像提取的概念架构
class UserProfile:
    def __init__(self, user_id):
        self.user_id = user_id
        self.static_attrs = {}     # 注册信息
        self.behavior_features = {} # 行为特征
        self.preference_model = {}  # 偏好模型
    
    def update_from_interaction(self, interaction):
        """每次交互后更新画像"""
        # 1. 提取知识水平信号
        complexity = estimate_query_complexity(interaction.query)
        self.behavior_features['knowledge_level'] = \
            ema(self.behavior_features.get('knowledge_level', 0.5), 
                complexity, alpha=0.1)
        
        # 2. 提取风格偏好
        if interaction.feedback == 'positive':
            style = extract_style(interaction.response)
            self.preference_model[style] = \
                self.preference_model.get(style, 0) + 1
        
        # 3. 提取活跃模式
        self.behavior_features['active_hours'] = \
            update_time_distribution(interaction.timestamp)
    
    def to_prompt_context(self):
        """转化为可注入 Prompt 的上下文"""
        return f"""
        用户画像:
        - 知识水平: {self.knowledge_level_label}
        - 偏好风格: {self.preferred_style}
        - 常用场景: {self.top_use_cases}
        """
```

### 5.2 个性化推荐在 AI 应用中的应用

个性化不只是"称呼用户名"这么简单，它影响 AI 系统的多个层面：

| 个性化维度 | 实现方式 | 效果 |
|-----------|---------|------|
| **回答风格** | 根据用户偏好调整详略、专业度 | 满意度提升 15-25% |
| **检索范围** | 根据用户角色过滤/加权检索结果 | 相关性提升 20-30% |
| **工具推荐** | 根据使用历史优先展示常用功能 | 效率提升 10-20% |
| **主动建议** | 根据行为模式预判需求 | 参与度提升 30-40% |
| **记忆召回** | 根据历史对话提供连续性体验 | 留存率提升 20-30% |

### 5.3 隐私保护的个性化

个性化与隐私是天然矛盾的。技术方案在两者间寻找平衡：

#### 联邦学习 (Federated Learning)

```
         ┌──────────┐
         │ 中央服务器 │  ← 只收集模型梯度，不收集原始数据
         └─────┬────┘
               │ 聚合梯度
     ┌─────────┼─────────┐
     │         │         │
┌────▼───┐┌───▼────┐┌───▼────┐
│ 设备 A  ││ 设备 B  ││ 设备 C  │  ← 数据留在本地
│ 本地数据 ││ 本地数据 ││ 本地数据 │
│ 本地训练 ││ 本地训练 ││ 本地训练 │
└────────┘└────────┘└────────┘
```

**优势**: 数据不出域，满足隐私法规
**挑战**: 通信成本高、Non-IID 数据导致收敛困难、模型精度可能略降

#### 差分隐私 (Differential Privacy)

```
核心思想: 在数据/模型中添加精心设计的噪声，
         使得无法从输出反推单个用户的信息。

数学定义:
对于所有仅在一个样本上不同的数据集 D 和 D'，
以及所有输出集合 S:

    Pr[M(D) ∈ S] ≤ e^ε · Pr[M(D') ∈ S] + δ

ε (隐私预算): 越小越隐私，但数据可用性越低
δ (松弛参数): 通常设为 10^-5 量级

实践中的 trade-off:
ε = 1:  强隐私保护，模型精度下降明显 (~5-10%)
ε = 10: 弱隐私保护，模型精度几乎不受影响
```

#### 本地化存储方案

**最实用的隐私方案**: 将用户画像存储在**用户本地/设备端**，服务端只做匿名化的聚合分析。

```
用户设备端:
  ├─ 完整用户画像 (加密存储)
  ├─ 交互历史 (本地数据库)
  └─ 个性化模型参数 (本地 adapter/LoRA)

服务端:
  ├─ 匿名聚合统计 (不含个人信息)
  ├─ 通用基础模型
  └─ 群体画像 (角色/行业级别)

交互时:
  请求 = 用户查询 + 本地画像摘要 (用户授权发送)
  → 服务端基于画像摘要个性化回答
  → 画像摘要不存储，用后即删
```

### 5.4 长期记忆与个性化的结合

个性化的最高形态是 AI 拥有**关于用户的长期记忆**：

```
长期记忆层次:

第一层: 事实记忆
    "用户叫张三，是后端开发者，用 Go 语言"
    → 存储方式: 结构化 KV 存储
    → 更新策略: 用户主动告知时更新

第二层: 偏好记忆
    "用户喜欢简洁回答，偏好代码而非文字描述"
    → 存储方式: 向量化的偏好特征
    → 更新策略: 从反馈中持续学习

第三层: 情景记忆
    "上次用户问了关于微服务架构的问题，推荐了 Go-Kit"
    → 存储方式: 向量数据库 (检索)
    → 更新策略: 每次对话后存入

第四层: 元认知记忆
    "用户在 AI Agent 话题上是高级水平，在前端话题上是入门水平"
    → 存储方式: 用户画像的能力矩阵
    → 更新策略: 定期从交互中推断更新
```

> **📌 面试高频**: "如何在 AI 产品中实现个性化？" → 分三个层次回答：1) Prompt 注入用户画像（最快最简单）；2) 检索个性化（根据画像调整检索策略）；3) 模型级个性化（用户数据微调 adapter）。并说明隐私保护方案。

## 六、数据版本管理

### 6.1 为什么数据需要版本管理

代码有 Git，但数据呢？在 AI 系统中，**数据变更对模型效果的影响可能比代码变更更大**。

```
没有数据版本管理时:
  - "上周模型效果更好" → 是因为换了数据集？换了哪个？
  - "线上出现幻觉" → 是新训练数据引入了错误？哪条？
  - "能回滚到上周的版本吗？" → 训练数据都覆盖了...
  
有了数据版本管理:
  - 每次训练记录使用的数据集版本
  - 数据集变更有完整 diff
  - 可以精确回滚到任何历史版本
  - 数据变更可以做 code review
```

### 6.2 核心工具对比

| 工具 | 定位 | 核心思想 | 适用场景 | 存储后端 |
|------|------|---------|---------|---------|
| **DVC** | Git for Data | 用 Git 管元数据，远程存储管实际数据 | ML 项目，中小团队 | S3/GCS/Azure/SSH |
| **LakeFS** | 数据湖版本管理 | 在对象存储上实现 Git 语义 | 大数据平台，数据湖 | S3 兼容存储 |
| **Delta Lake** | ACID 数据湖 | Spark 生态，事务支持 | Spark 数据管道 | HDFS/S3 |
| **HF Datasets** | ML 数据集管理 | 社区+版本+加载器一体 | 公开数据集，标准化 | Hugging Face Hub |
| **Pachyderm** | 数据管道版本管理 | 数据+处理逻辑一起版本化 | 复杂 ETL 管道 | K8s + 对象存储 |

#### DVC 详解

DVC (Data Version Control) 是最常用的 ML 数据版本管理工具：

```bash
# 初始化 DVC
dvc init

# 追踪大文件/目录
dvc add data/training_set_v1.parquet
# 生成 data/training_set_v1.parquet.dvc (元数据文件)
# 实际数据被移到 .dvc/cache

# 将元数据提交到 Git
git add data/training_set_v1.parquet.dvc .gitignore
git commit -m "Add training set v1"

# 推送数据到远程存储
dvc remote add myremote s3://my-bucket/dvc-storage
dvc push

# 切换到不同版本的数据
git checkout v1.0
dvc checkout
# 数据自动恢复到 v1.0 对应的版本！

# 查看数据管道
dvc dag  # 可视化数据管道 DAG
```

**DVC 的核心概念**:

```
项目根目录/
├── .git/           ← Git 管理代码和元数据
├── .dvc/           ← DVC 缓存
│   └── cache/      ← 实际数据文件的内容寻址存储
├── data/
│   ├── raw/
│   │   └── crawled_data.csv.dvc  ← 元数据 (Git 追踪)
│   └── processed/
│       └── train.parquet.dvc      ← 元数据 (Git 追踪)
├── dvc.yaml        ← 数据管道定义
├── dvc.lock        ← 管道执行锁 (输入/输出的 hash)
└── params.yaml     ← 参数文件
```

### 6.4 数据集的 CI/CD

像代码一样对数据做自动化质量检查：

```yaml

  push:
    paths: ['data/**']

jobs:
  data-quality-check:
    runs-on: ubuntu-latest

      - name: Quality metrics
        run: python scripts/check_quality.py data/train.parquet

      - name: Contamination check
```

**关键质量检查项**:

| 检查项 | 阈值 | 说明 |
|--------|------|------|
| Schema 一致性 | 100% 合规 | 字段名、类型必须与 Schema 匹配 |
| 空值比例 | < 5% | 关键字段不允许空值 |
| 重复率 | < 1% | 去重后残留的重复 |
| 文本长度分布 | KS-test p > 0.05 | 与基线分布无显著差异 |
| 数据污染 | 0 overlap | 训练集与测试集零重叠 |
| 分布偏移 (PSI) | < 0.1 | 与基线数据的分布偏移在可控范围 |
| PII 残留 | 0 | 脱敏后不应残留个人信息 |

## 七、数据隐私与安全

### 7.1 PII 检测与脱敏

PII (Personally Identifiable Information) 是数据安全的**第一道防线**。

**常见 PII 类型及检测方案**:

| PII 类型 | 检测方法 | 脱敏策略 |
|---------|---------|---------|
| 姓名 | NER 模型 + 字典匹配 | 替换为 `[NAME_1]` |
| 手机号 | 正则表达式 `1[3-9]\d{9}` | 替换为 `138****1234` |
| 身份证号 | 正则 + 校验位验证 | 替换为 `[ID_NUMBER]` |
| 邮箱 | 正则 `\w+@\w+\.\w+` | 替换为 `***@example.com` |
| 银行卡号 | Luhn 算法 + 正则 | 替换为 `[CARD_NUMBER]` |
| 地址 | NER + 地址解析 | 泛化到城市级 |
| IP 地址 | 正则 | 替换为 `[IP_ADDRESS]` |

**多层脱敏架构**:

```
原始数据输入
    │
    ├─ 第一层: 正则规则 (速度快, 覆盖标准格式)
    │   └─ 电话、邮箱、身份证号、银行卡
    │
    ├─ 第二层: NER 模型 (处理非标准表达)
    │   └─ "我叫张三丰, 住在深圳南山区..."
    │       → "我叫[NAME], 住在[LOCATION]..."
    │
    ├─ 第三层: 上下文分析 (推理型 PII)
    │   └─ "我就是昨天打电话那个人" → 上下文关联到具体身份
    │
    └─ 第四层: 人工审核 (高风险数据抽检)
        └─ 对脱敏结果进行质量检查

脱敏后数据输出
```

**工具推荐**:
- **Microsoft Presidio**: 开源 PII 检测与脱敏框架，支持多语言
- **SpaCy + 自定义 NER**: 灵活，适合定制化需求
- **Google DLP API**: 云服务，支持 100+ 种 PII 类型

### 7.3 防止模型记忆训练数据

LLM 有"记忆"训练数据的倾向，可能在推理时**泄露**训练数据中的敏感信息。

**攻击场景**:
```
攻击者 Prompt: "请背诵以下电话号码开头的后续内容: 138..."
LLM 回答: "13812345678, 这是张三的电话..."  ← 从训练数据中"回忆"出来
```

**防御措施**:

| 防御层 | 措施 | 说明 |
|--------|------|------|
| 数据层 | 训练前 PII 脱敏 | 源头上消除敏感信息 |
| 训练层 | 差分隐私训练 (DP-SGD) | 在梯度中加噪声，限制单条数据的影响 |
| 模型层 | 记忆检测 | 用提取攻击（extraction attack）检测模型是否记住了训练数据 |
| 推理层 | 输出过滤 | 检测输出中是否包含 PII 模式，拦截 |
| 监控层 | 异常检测 | 检测可疑的数据提取尝试 |

**DP-SGD (差分隐私随机梯度下降)**:

```
标准 SGD:
    θ ← θ - η · ∇L(θ, batch)

DP-SGD:
    1. 计算每个样本的梯度: g_i = ∇L(θ, x_i)
    2. 裁剪梯度范数: g̃_i = g_i · min(1, C/‖g_i‖)  (C 是裁剪阈值)
    3. 加噪声: g̃ = (1/B)Σg̃_i + N(0, σ²C²I)
    4. 更新: θ ← θ - η · g̃

效果: 数学上保证模型无法"精确记住"任何单条训练数据
代价: 训练速度慢 2-10 倍，模型精度下降 1-5%
```

### 7.4 跨境数据传输的合规要求

在全球化 AI 项目中，数据跨境是**高频雷区**：

```
中国 → 海外:
├─ 个人信息保护法 第38条: 需通过安全评估/标准合同/认证
├─ 数据安全法: 重要数据需要安全评估
├─ 实操: 
│   ├─ < 10万人个人信息 → 标准合同备案
│   ├─ > 10万人 或 敏感信息 > 1万人 → 安全评估
│   └─ 重要数据 → 必须安全评估
└─ 时间: 安全评估通常需要 2-6 个月

欧盟 → 非欧盟:
├─ GDPR 第44-49条: 充分性决定 / SCCs / BCRs
├─ 中国不在"充分性"名单中
└─ 需要签署标准合同条款 (SCCs)

美国:
├─ 无联邦统一法规
├─ 各州法规不同 (加州最严)
└─ 特定行业有专门法规 (HIPAA-医疗, GLBA-金融)
```

**合规落地的技术方案**:

```
方案 1: 数据本地化
    ├─ 在每个地区部署独立的数据存储
    ├─ 本地训练，只传输模型参数
    └─ 适用: 严格合规要求的场景

方案 2: 联邦学习
    ├─ 数据不出域，只交换梯度
    ├─ 加密聚合 (Secure Aggregation)
    └─ 适用: 多地区协作训练

方案 3: 数据脱敏后传输
    ├─ 完全脱敏后的数据可能不受限
    ├─ 需要法律确认脱敏标准是否足够
    └─ 适用: 数据分析、统计用途

方案 4: 差分隐私导出
    ├─ 对统计结果加噪后导出
    ├─ 数学上可证明不泄露个人信息
    └─ 适用: 聚合分析、报表
```

> **📌 面试高频**: "你们的数据是怎么处理隐私合规的？" → 从三个层面回答：1) 数据分类分级（个人信息/敏感信息/重要数据）；2) 技术措施（PII 脱敏、加密存储、访问控制）；3) 制度措施（数据安全评估、合规审查流程）。

|------|---------|
| MinHash 去重原理 | Jaccard 相似度的近似估计，通过 LSH 加速相似文档查找 |
| 困惑度过滤的直觉 | 用语言模型衡量文本"自然度"，过滤机器生成/乱码/模板文本 |
| 大规模爬虫架构 | Scrapy 集群 + Redis 调度 + Bloom Filter 去重 + 多级过滤 |
| 数据质量对模型的影响 | Scaling Law 前提是高质量数据；10% 噪声数据可能让模型效果下降 20-30% |

### 数据标注

| 考点 | 关键回答 |
|------|---------|
| Cohen's Kappa 含义 | 排除随机一致性后的标注一致性指标，> 0.6 可接受，> 0.8 理想 |
| 主动学习三大策略 | 不确定性采样、多样性采样、混合策略 |
| 偏好标注的挑战 | 主观性高、位置偏见、一致性差、需要领域知识 |
| 标注质量控制 | 事前(指南+培训) + 事中(金标准+实时监控) + 事后(交叉验证+审核) |

### 合成数据

| 考点 | 关键回答 |
|------|---------|
| Self-Instruct vs Evol-Instruct | Self-Instruct 从零生成新指令；Evol-Instruct 进化现有指令的复杂度 |
| Model Collapse | 递归使用合成数据训练会导致多样性崩塌，需混入真实数据 |
| 合成数据能否替代真实数据 | 不能完全替代，解决"量"和"多样性"，但"分布真实性"仍需真实数据 |
| 合成:真实的最佳比例 | 取决于场景，通用 SFT 约 7:3，偏好对齐约 4:6 |

### 数据飞轮

| 考点 | 关键回答 |
|------|---------|
| 数据飞轮核心理念 | 使用→数据→优化→更好体验→更多使用 的正反馈循环 |
| 冷启动策略 | 四阶段：公开数据 → 合成数据 → 种子用户 → 全量飞轮 |
| 显式 vs 隐式反馈 | 显式(👍👎)信号清晰但量少；隐式(行为)量大但噪声高 |
| A/B 测试要点 | 按用户ID Hash分流、注意新奇效应、样本量要足够 |

### 数据隐私

| 考点 | 关键回答 |
|------|---------|
| 差分隐私核心思想 | 添加噪声使得无法从输出反推单个用户信息，ε 控制隐私-可用性权衡 |
| 联邦学习 | 数据不出域，只交换模型梯度，中央聚合 |
| 防止模型记忆训练数据 | PII脱敏 + DP-SGD + 输出过滤 + 记忆检测 |
| GDPR/个保法核心区别 | GDPR 强调同意+DPO；中国个保法强调安全评估+跨境限制 |

### 数据版本管理

| 考点 | 关键回答 |
|------|---------|
| DVC 核心原理 | Git 管元数据(.dvc 文件)，远程存储管实际数据，通过 hash 关联 |
| 数据血缘的价值 | 模型出问题时可以追溯到源数据，支持复现和回滚 |
| 数据 CI/CD | Schema验证 + 质量指标检查 + 分布偏移检测 + 污染检查 |
| Data Card | 数据集的"身份证"，包含来源、构成、质量、限制、伦理信息 |

## 第十四章 面试深度问答

### Q1: 大模型训练数据是如何准备的？描述完整流程

**参考解答：**

大模型训练数据准备是一个复杂的系统工程，通常包含以下阶段：

**1. 数据采集**
- Web爬取：CommonCrawl等公开爬虫数据，通常数十TB
- 书籍/论文：Gutenberg、arXiv、Semantic Scholar
- 代码：GitHub公开仓库
- 对话：Reddit、Stack Overflow
- 多语言：各语种维基百科、新闻、论坛

**2. 数据清洗（最关键步骤）**
- 格式标准化：HTML解析、编码统一
- 去重：URL级精确去重 + MinHash近似去重（通常去除30-60%数据）
- 质量过滤：用分类器评分、基于启发式规则过滤低质量内容
- 安全过滤：PII脱敏、有害内容移除、NSFW过滤

**3. 数据混合**
- 按领域分桶（代码/学术/网页/书籍/对话）
- 确定混合比例（这是影响模型能力的关键决策）
- 通常代码10-15%、学术10%、书籍15%、高质量网页50%

**4. Token化与打包**
- 训练/选用Tokenizer（BPE为主流）
- 文档拼接到最大序列长度
- 存储为高效二进制格式

**5. 持续迭代**
- 训练过程中监控数据质量信号
- 根据模型弱点定向补充数据
- 数据飞轮：部署后用户反馈回流

### Q2: Feature Store解决什么问题？为什么需要它？

**参考解答：**

Feature Store解决的核心问题是**特征管理的复杂性和不一致性**：

1. **训练-推理偏差（Training-Serving Skew）**：训练时用的特征和线上推理时用的特征计算逻辑不一致，导致模型效果下降。Feature Store确保特征定义一处编写、到处使用。

2. **特征复用**：不同团队/模型经常需要相同的特征（如用户画像特征），没有Feature Store时每个团队重复计算，浪费资源。

3. **特征时间旅行**：训练时需要获取历史某个时间点的特征值（Point-in-Time Join），防止数据泄露。Feature Store维护特征的时间版本。

4. **在线/离线一致性**：离线用Spark SQL计算的特征，在线需要用Redis低延迟服务，两者逻辑需要统一管理。

5. **特征发现与文档**：团队间了解有哪些可用特征、特征的含义和质量，Feature Store提供特征目录和搜索功能。

**选型建议：**
- 早期/小团队：不需要Feature Store，简单的缓存+数据库够用
- 中期（5+模型在线）：Feast（开源轻量）或自建简单Feature Store
- 大规模（20+模型）：Tecton/Hopsworks或自研平台

### Q3: 如何评估训练数据质量？

**参考解答：**

训练数据质量评估需要从多个维度进行：

| 维度 | 评估方法 | 度量指标 |
|------|---------|---------|
| 准确性 | 人工抽检、交叉验证 | 错误率 < 5% |
| 完整性 | 字段填充率统计 | 必填字段100%，可选字段>80% |
| 一致性 | 跨源数据对比 | 冲突率 < 1% |
| 多样性 | 分布分析、N-gram分析 | 覆盖目标场景90%+ |
| 去重率 | MinHash/SimHash检测 | 重复率 < 5%（去重后） |
| 安全性 | 毒性分类器+PII检测 | 有害内容 < 0.1% |
| 均衡性 | 类别分布分析 | 少数类占比合理 |
| 新鲜度 | 时间戳统计 | 90%数据在可接受时间范围内 |

关键工具：
- Great Expectations：数据断言和自动化测试
- Pandas Profiling / ydata-profiling：数据集概况报告
- Cleanlab：自动检测标注错误
- 自定义质量评分模型（用LLM打分）

### Q4: 如何设计一个处理TB级文本数据的去重系统？

**参考解答：**

TB级文本去重系统的设计需要分层处理：

**第一层：精确去重**
- URL/文档ID去重：简单哈希，O(1)查找
- 全文MD5/SHA256去重：针对完全相同的文档
- 工具：Bloom Filter（节省内存）或Hash Set

**第二层：近似去重（MinHash LSH）**
- 步骤：
  1. 文档→N-gram集合（通常5-gram）
  2. 计算MinHash签名（128个hash函数）
  3. LSH分桶：将签名分为b个band，每band r行
  4. 同一桶内的文档对计算精确Jaccard
  5. Jaccard > 0.8判为重复
- 分布式实现：Spark + 自定义LSH或使用datasketch库
- 内存优化：每个文档只需128×4=512 bytes签名

**第三层：段落级去重**
- 解决文档不完全重复但包含大段相同内容的情况
- 方法：SimHash句子级 → 找重复段落 → 保留原创内容更多的版本

**工程考量：**
- 计算量：O(N)签名 + O(N*b)分桶，比O(N²)暴力对比高效得多
- 分布式：数据按hash分片到不同worker
- 增量：新文档只需计算签名，与已有索引对比

### Q5: 数据飞轮是什么？怎么建设？

**参考解答：**

数据飞轮是一个正向循环系统：更多用户产生更多数据 → 更好的模型 → 更好的产品 → 吸引更多用户 → 更多数据。

**建设步骤：**

1. **数据采集层**：埋点收集用户交互数据（点击/停留/反馈）
2. **自动化标注**：利用用户隐式反馈（如用户编辑AI输出 → 暗示原始输出不够好）
3. **模型持续训练**：定期用新数据微调模型
4. **A/B测试验证**：确认新模型确实更好
5. **灰度发布**：逐步放量新模型
6. **监控告警**：持续监控模型质量指标

**关键设计原则：**
- 自动化：全链路尽量自动化，减少人工干预
- 反馈闭环：用户反馈必须回流到训练数据
- 质量保障：自动化不等于没有质量控制
- 渐进式：先建小飞轮（单一场景），再扩展到大飞轮

**注意事项：**
- 避免反馈偏差（曝光偏差、位置偏差）
- 防止模型自我强化（推荐→点击→推荐更多类似→过度集中）
- 合成数据比例控制（防止模型坍塌）
- 隐私合规（用户数据使用需经同意）

### H.1 使用Polars进行高效数据清洗

```python
import polars as pl

# 读取大规模CSV（比Pandas快5-10倍）
df = pl.scan_csv("data/raw/*.csv")  # 惰性求值

# 数据清洗Pipeline
cleaned = (
    df
    # 去除空值
    .filter(pl.col("text").is_not_null())
    .filter(pl.col("text").str.len_chars() > 50)
    # 文本标准化
    .with_columns([
        pl.col("text")
          .str.strip_chars()
          .str.replace_all(r"\s+", " ")
          .alias("text_clean"),
        pl.col("timestamp").cast(pl.Datetime).alias("ts"),
    ])
    # 去重
    .unique(subset=["text_clean"])
    # 添加统计特征
    .with_columns([
        pl.col("text_clean").str.len_chars().alias("text_len"),
        pl.col("text_clean").str.count_matches(r"\w+").alias("word_count"),
    ])
    # 质量过滤
    .filter(pl.col("word_count") > 10)
    .filter(pl.col("text_len") < 50000)
    # 排序
    .sort("ts", descending=True)
)

# 执行并写入Parquet
cleaned.collect().write_parquet("data/cleaned/output.parquet")
```

### H.2 MinHash去重实现

```python
from datasketch import MinHash, MinHashLSH
import hashlib

def create_minhash(text, num_perm=128):
    """创建文档的MinHash签名"""
    m = MinHash(num_perm=num_perm)
    # 5-gram shingling
    for i in range(len(text) - 4):
        shingle = text[i:i+5]
        m.update(shingle.encode('utf-8'))
    return m

def deduplicate_documents(documents, threshold=0.8):
    """大规模文档去重"""
    # 创建LSH索引
    lsh = MinHashLSH(threshold=threshold, num_perm=128)
    
    unique_docs = []
    duplicates = set()
    
    for idx, doc in enumerate(documents):
        if idx in duplicates:
            continue
            
        mh = create_minhash(doc['text'])
        doc_id = f"doc_{idx}"
        
        # 查询相似文档
        result = lsh.query(mh)
        
        if len(result) == 0:
            # 无重复，加入索引
            lsh.insert(doc_id, mh)
            unique_docs.append(doc)
        else:
            duplicates.add(idx)
    
    print(f"去重前: {len(documents)}, 去重后: {len(unique_docs)}")
    print(f"去重率: {1 - len(unique_docs)/len(documents):.2%}")
    return unique_docs
```

```python
        column="text", min_value=0.95
    )
)

# 类别分布检查

    gx.expectations.ExpectColumnDistinctValuesToBeInSet(
```

### H.4 Airflow DAG数据管道模板

```python
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

default_args = {
    'owner': 'data-team',
    'depends_on_past': False,
    'retries': 2,
    'retry_delay': timedelta(minutes=5),
}

with DAG(
    'training_data_pipeline',
    default_args=default_args,
    description='AI Training Data Pipeline',
    schedule_interval='0 2 * * *',  # 每天凌晨2点执行
    start_date=datetime(2025, 1, 1),
    catchup=False,
    tags=['data-engineering', 'ml'],
) as dag:

    # Step 1: 数据采集
    collect = PythonOperator(
        task_id='collect_data',
        python_callable=collect_data_func,
    )

    # Step 2: 数据清洗
    clean = PythonOperator(
        task_id='clean_data',
        python_callable=clean_data_func,
    )

    # Step 3: 质量检查
    quality_check = PythonOperator(
        task_id='quality_check',
        python_callable=run_quality_checks,
    )

    # Step 4: 去重
    dedup = PythonOperator(
        task_id='deduplicate',
        python_callable=deduplicate_func,
    )

    # Step 5: 写入数据湖
    write = BashOperator(
        task_id='write_to_lake',
        bash_command='spark-submit write_to_delta.py',
    )

    # Step 6: 更新数据版本
    version = PythonOperator(
        task_id='update_version',
        python_callable=update_dvc_version,
    )

    # DAG依赖
    collect >> clean >> quality_check >> dedup >> write >> version
```
