# 📊 数据工程与数据飞轮 — Data Engineering & Data Flywheel

## 概述

数据是 AI 系统的燃料。数据工程关注如何高效地采集、清洗、标注、存储和管理数据；数据飞轮则是通过"用户使用 → 收集反馈 → 改进模型 → 更好体验"的闭环，实现 AI 系统的持续自我改进。高质量的数据工程能力是构建优秀 AI 系统的基石。

## 核心知识体系

### 1. 数据采集与清洗

- **数据来源**：
  - 公开数据集：Hugging Face Datasets、Common Crawl、Wikipedia
  - 网络爬虫：Scrapy、Crawl4AI、Firecrawl
  - API 数据：社交媒体、新闻、学术论文
  - 企业内部数据：数据库、文档、日志
- **数据清洗**：
  - 去重：MinHash、SimHash、精确去重
  - 质量过滤：困惑度过滤、分类器过滤、规则过滤
  - 有害内容过滤：毒性检测、PII（个人身份信息）脱敏
  - 格式标准化：编码统一、格式转换、结构化提取
- **大规模数据处理**：
  - PySpark / Spark：分布式数据处理
  - Dask：Python 原生并行计算
  - Ray Data：分布式数据管道

### 2. 数据标注

#### 2.1 标注方法
- **人工标注**：
  - 标注平台：Label Studio、Prodigy、Scale AI、Amazon SageMaker Ground Truth
  - 标注规范设计：标注指南、质量标准、一致性检查
  - 标注团队管理：培训、质检、争议解决
- **LLM 辅助标注** ⭐：
  - 用大模型生成初始标注，人工审核修正
  - 主动学习（Active Learning）：优先标注模型最不确定的样本
  - 自动标注 + 人工抽检的混合模式
- **众包标注**：
  - Amazon Mechanical Turk
  - 质量控制：金标准题、交叉验证、多人标注取共识

#### 2.2 标注类型
| 任务 | 标注内容 |
|------|---------|
| 指令微调 | 指令-回答对（Instruction-Response） |
| 偏好对齐 | 回答排序/偏好对（Chosen-Rejected） |
| NER | 实体边界与类型标注 |
| 关系抽取 | 实体间关系标注 |
| 文本分类 | 类别标签 |
| RAG 评估 | 查询-文档相关性标注 |

### 3. 合成数据生成 ⭐

- **指令数据合成**：
  - Self-Instruct：用 LLM 自动生成指令数据
  - Evol-Instruct（WizardLM）：渐进式增加指令复杂度
  - Alpaca 方法：用 GPT-4 生成高质量指令数据
- **偏好数据合成**：
  - 用不同模型生成多个回答，人工/自动排序
  - Constitutional AI 方法：用原则引导生成正负样本
- **领域数据合成**：
  - 基于种子数据扩展：少量真实数据 → LLM 扩写
  - 角色扮演生成：让 LLM 扮演特定角色生成数据
  - 多样性控制：确保合成数据覆盖足够多的场景
- **合成数据质量控制**：
  - 去重与去污染
  - 多样性评估
  - 人工抽样审核
  - 下游任务效果验证

### 4. 数据飞轮（Data Flywheel）⭐

```
用户使用 AI 产品
    → 收集用户交互数据（查询、反馈、行为）
    → 数据清洗与标注
    → 模型微调 / RAG 知识库更新
    → 产品体验提升
    → 更多用户使用（正向循环）
```

- **反馈收集机制**：
  - 显式反馈：👍👎 评分、文字反馈、纠错
  - 隐式反馈：点击率、停留时间、重新生成次数、复制行为
  - 对话级反馈 vs 回答级反馈
- **反馈数据利用**：
  - 正向反馈 → 高质量训练数据
  - 负向反馈 → 识别模型弱点，定向改进
  - 用户行为分析 → 产品优化方向
- **持续改进流程**：
  - 定期模型更新（周期性微调）
  - 知识库增量更新
  - A/B 测试验证改进效果
  - 自动化数据管道（ETL）

### 5. 在线学习与个性化 ⭐

> 数据飞轮的终极目标是让 Agent 越用越好、越用越懂用户。但实时个性化的工程实现远比想象复杂。

- **用户画像构建**：
  - 从交互历史中提取用户特征：
  | 维度 | 提取方式 | 用途 |
  |------|---------|------|
  | 知识水平 | 分析用户提问的专业程度 | 调整回答的深度和术语使用 |
  | 沟通风格偏好 | 统计用户对不同风格回答的反馈 | 调整回答的语气和详略 |
  | 常用功能 | 统计工具调用频率 | 优先展示常用功能 |
  | 历史问题模式 | 聚类分析历史查询 | 预测用户意图 |
  | 满意度趋势 | 追踪反馈评分变化 | 识别体验退化 |

- **个性化 Prompt 注入**：
  ```
  System Prompt = 基础指令 + 用户画像摘要 + 场景指令
  
  用户画像摘要示例：
  "该用户是高级开发者，偏好简洁技术性回答，常用 Python，
   历史交互中对代码示例的满意度最高。"
  ```

- **个性化检索**：
  - 根据用户历史行为调整检索权重（如：技术用户优先返回技术文档）
  - 根据用户部门/角色过滤检索范围
  - 基于用户反馈的相关性反馈（Relevance Feedback）优化检索

- **A/B 测试框架**：
  - 对不同用户群体做个性化策略的 A/B 测试
  - 分流策略：按用户 ID 哈希分流，确保同一用户始终在同一组
  - 核心指标：用户满意度、任务完成率、会话轮次、Token 消耗
  - 统计显著性：确保样本量足够，避免过早下结论

- **冷启动问题**：
  | 阶段 | 策略 |
  |------|------|
  | 全新用户 | 使用默认画像（基于用户角色/部门的群体画像） |
  | 少量交互（1-5 轮） | 快速特征提取（问题复杂度、语言偏好） |
  | 中等交互（5-20 轮） | 初步画像建立，开始个性化 |
  | 充分交互（20+ 轮） | 完整画像，深度个性化 |

- **隐私边界**：
  - 个性化与隐私保护的平衡
  - 用户数据保留策略：保留多久？用来做什么？用户能否删除？
  - 透明度：告知用户哪些数据被用于个性化
  - 合规要求：GDPR 的"被遗忘权"、数据最小化原则
  - 技术方案：差分隐私、联邦学习、本地化存储

### 6. 数据版本管理

- **工具**：
  - DVC（Data Version Control）：Git for Data
  - LakeFS：数据湖版本管理
  - Delta Lake：ACID 事务支持的数据湖
- **最佳实践**：
  - 数据集版本化与可追溯
  - 训练数据与模型版本关联
  - 数据血缘（Data Lineage）追踪

### 6. 数据隐私与安全

- **PII 检测与脱敏**：姓名、电话、身份证号等敏感信息处理
- **差分隐私（Differential Privacy）**：在数据中添加噪声保护个体隐私
- **联邦学习**：数据不出域的协作训练
- **数据访问控制**：RBAC、数据分级分类
