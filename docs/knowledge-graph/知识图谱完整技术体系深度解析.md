## 一、知识图谱基础

### 1.1 核心数据模型：三元组与图

知识图谱的本质是一个**有向标记图**（Directed Labeled Graph），其中：
- **节点（Node）** = 实体（Entity）或概念
- **边（Edge）** = 关系（Relation）
- 知识以 **三元组（Triple）** 形式表达：`(头实体, 关系, 尾实体)` 或 `(Subject, Predicate, Object)`

```
示例三元组：
(张三, 就职于, 腾讯)
(腾讯, 总部位于, 深圳)
(深圳, 位于, 广东省)
(张三, 毕业于, 中山大学)
(中山大学, 类型, 985高校)
```

这些三元组连接起来就形成了一张知识网络：

```
张三 ──就职于──→ 腾讯 ──总部位于──→ 深圳 ──位于──→ 广东省
 │                                    
 └──毕业于──→ 中山大学 ──类型──→ 985高校
```

**核心直觉**：三元组是知识图谱的"原子"，就像关系数据库中的"行"一样不可再分。但与表格不同，三元组天然支持**关系推理**——沿着边遍历就能发现隐含知识。

### 1.2 两大图模型：RDF vs 属性图

知识图谱有两种主流数据模型，选择哪种直接决定了技术栈走向：

```
┌────────────────────────────────────────────────────────────────┐
│                      知识图谱数据模型                           │
├──────────────────────────┬─────────────────────────────────────┤
│         RDF 模型          │           属性图模型                │
│  (Resource Description    │     (Property Graph Model)         │
│   Framework)              │                                    │
├──────────────────────────┼─────────────────────────────────────┤
│ • W3C 标准               │ • 事实标准（Neo4j 等推动）          │
│ • 严格三元组：S-P-O       │ • 节点和边都可携带属性              │
│ • URI 标识一切            │ • 更直观、更灵活                    │
│ • 配合 OWL 做本体推理     │ • 工程实践中更主流                  │
│ • 查询语言：SPARQL        │ • 查询语言：Cypher/Gremlin          │
│ • 适合：语义网、开放数据   │ • 适合：企业应用、推荐系统          │
└──────────────────────────┴─────────────────────────────────────┘
```

**RDF 模型详解**：

```turtle
# RDF Turtle 语法示例
@prefix ex: <http://example.org/> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .

ex:张三 ex:就职于 ex:腾讯 .
ex:张三 ex:年龄 "29"^^xsd:integer .
ex:腾讯 rdfs:label "腾讯控股有限公司" .
```

RDF 的核心特点：
- **一切皆 URI**：每个实体、关系都有全球唯一标识符
- **严格三元组**：边不能有属性（要表达"2022年入职"，需要引入中间节点——**具体化/Reification**）
- **开放世界假设**：未声明的不代表为假
- **本体推理**：配合 OWL/RDFS 可以做逻辑推理（如子类推理、传递性推理）

**属性图模型详解**：

```
// 属性图（Cypher 语法）
CREATE (zhangsan:Person {name: "张三", age: 29})
CREATE (tencent:Company {name: "腾讯", founded: 1998})
CREATE (zhangsan)-[:WORKS_AT {since: 2022, role: "工程师"}]->(tencent)
```

属性图的核心优势：
- **边可以有属性**：`since: 2022` 直接挂在边上，无需具体化
- **更符合直觉**：与人类画关系图的方式一致
- **查询更简洁**：Cypher 比 SPARQL 对开发者更友好
- **工程实践主流**：Neo4j、NebulaGraph 等都采用属性图

> 💡 **面试考点**：RDF vs 属性图的核心区别？RDF 中如何表达边的属性？答案：RDF 用具体化（Reification），引入中间节点把三元组变成"关于三元组的描述"；属性图原生支持边属性。

### 1.3 本体与模式设计

**本体（Ontology）** 是知识图谱的"Schema"，定义了：
- **类层次**：Person 是 Entity 的子类，Student 是 Person 的子类
- **关系定义**：`works_at` 的定义域（domain）是 Person，值域（range）是 Organization
- **约束规则**：一个人只能有一个出生日期（函数性约束）
- **公理**：`父亲` 关系的 range 是 `男性`

本体设计的层次：

```
┌─────────────────────────────────────────────┐
│  上层本体 (Upper Ontology)                    │
│  如 SUMO、DOLCE、BFO                         │
│  定义最通用的概念：实体、事件、时间、空间       │
├─────────────────────────────────────────────┤
│  领域本体 (Domain Ontology)                   │
│  如 SNOMED CT（医学）、FIBO（金融）           │
│  定义特定领域的概念和关系                      │
├─────────────────────────────────────────────┤
│  应用本体 (Application Ontology)              │
│  为特定应用定制的本体                          │
│  如"智能客服知识图谱的本体"                    │
└─────────────────────────────────────────────┘
```

**可扩展本体设计原则**：

1. **从核心实体开始**：先定义最重要的 5-10 种实体类型
2. **关系要有语义**：避免万能的 `related_to`，用精确的 `works_at`、`located_in`
3. **预留扩展点**：用继承机制，新类型作为已有类型的子类
4. **保持一致性**：命名规范统一（驼峰 vs 下划线）、语义不重叠
5. **参考已有标准**：schema.org、Dublin Core 等

### 1.4 知名知识图谱分析

| 知识图谱 | 规模 | 数据模型 | 特点 | 应用 |
|----------|------|---------|------|------|
| **Google Knowledge Graph** | 5000亿+ 事实 | 属性图 | 驱动搜索知识面板 | Google 搜索 |
| **Wikidata** | 1亿+ 实体 | RDF | 开放、多语言、社区维护 | Wikipedia、学术 |
| **DBpedia** | 数百万实体 | RDF | 从 Wikipedia 抽取 | 语义网研究 |
| **YAGO** | 1000万+ 实体 | RDF | 融合 Wikipedia + WordNet | 学术研究 |
| **CN-DBpedia** | 千万级实体 | 属性图 | 中文知识图谱 | 中文 NLP |
| **OpenKG** | 多领域 | 混合 | 中文开放知识图谱联盟 | 产业应用 |

**Google Knowledge Graph 设计分析**：
- 实体来源：Wikipedia、CIA World Factbook、Freebase 等
- 三层架构：实体层（who/what）→ 属性层（attributes）→ 关系层（connections）
- 核心能力：搜索结果右侧的知识面板、"People also ask"

**Wikidata 设计分析**：
- 数据模型：Item（实体）+ Property（属性）+ Statement（声明）
- 每个声明可以有 Qualifier（限定词）和 Reference（来源）
- 完全开放：CC0 许可证，任何人可编辑和使用

### 1.5 知识图谱 vs 关系数据库 vs 文档数据库

```
┌─────────────┬──────────────┬──────────────┬──────────────────┐
│   维度       │ 关系数据库    │ 文档数据库    │  知识图谱/图数据库 │
├─────────────┼──────────────┼──────────────┼──────────────────┤
│ 数据模型     │ 表格（行+列） │ JSON 文档    │ 节点+边+属性      │
│ Schema      │ 严格预定义    │ 灵活 Schema  │ 灵活，可选本体    │
│ 关系表达     │ 外键 JOIN    │ 嵌套/引用    │ 边（一等公民）     │
│ 多跳查询     │ 多次 JOIN    │ 多次查询     │ 图遍历（原生高效） │
│ 适用场景     │ 事务型数据    │ 半结构化数据  │ 复杂关系数据      │
│ 典型查询     │ "某人的订单"  │ "某篇文章"   │ "A与B的关系路径"  │
│ 性能瓶颈     │ 多表 JOIN    │ 深层关联查询  │ 超大规模全图计算  │
└─────────────┴──────────────┴──────────────┴──────────────────┘
```

**决策矩阵**：
- 数据关系简单、强事务需求 → **关系数据库**
- 数据半结构化、读多写少 → **文档数据库**
- 关系复杂、需要多跳推理、知识发现 → **知识图谱/图数据库**
- 很多实际系统是**混合架构**：关系数据库存核心交易数据，图数据库存关系网络

> 💡 **经典面试题**：社交网络中"找到我的朋友的朋友中也认识某人的人"，关系数据库需要 3 次 JOIN，图数据库只需一次 3 跳遍历，后者在关系稠密时性能优势可达数量级。

### 2.2 图查询语言深度解析

#### Cypher（Neo4j）

Cypher 是目前最流行的图查询语言，设计理念是"ASCII Art"——用字符画出图的模式：

```cypher
// 基本模式匹配：找到张三工作的公司
MATCH (p:Person {name: "张三"})-[:WORKS_AT]->(c:Company)
RETURN c.name

// 多跳查询：找到张三的同事的朋友
MATCH (p:Person {name: "张三"})-[:WORKS_AT]->(:Company)<-[:WORKS_AT]-(colleague)
      -[:FRIEND_OF]->(friend)
RETURN DISTINCT friend.name

// 可变长路径：找到两个实体之间的最短路径
MATCH path = shortestPath(
  (a:Person {name: "张三"})-[*..6]-(b:Person {name: "李四"})
)
RETURN path

// 聚合查询：每个公司有多少员工
MATCH (p:Person)-[:WORKS_AT]->(c:Company)
RETURN c.name, count(p) AS employee_count
ORDER BY employee_count DESC

// 创建节点和关系
CREATE (p:Person {name: "王五", age: 25})
CREATE (p)-[:GRADUATED_FROM {year: 2023}]->(:University {name: "清华大学"})

// 条件更新
MATCH (p:Person {name: "张三"})
SET p.title = "高级工程师"
RETURN p

// 子图模式：找到三角形关系（A认识B, B认识C, C也认识A）
MATCH (a)-[:KNOWS]->(b)-[:KNOWS]->(c)-[:KNOWS]->(a)
WHERE id(a) < id(b) AND id(b) < id(c)
RETURN a.name, b.name, c.name
```

**Cypher 核心语法元素**：

| 语法 | 含义 | 示例 |
|------|------|------|
| `()` | 节点 | `(p:Person)` |
| `-->` | 有向关系 | `(a)-->(b)` |
| `[:TYPE]` | 关系类型 | `[:WORKS_AT]` |
| `{key: val}` | 属性过滤 | `{name: "张三"}` |
| `*..n` | 可变长路径 | `[*1..3]` 1到3跳 |
| `MATCH` | 模式匹配 | 声明要找的图模式 |
| `WHERE` | 条件过滤 | 额外过滤条件 |
| `RETURN` | 返回结果 | 投影输出 |
| `WITH` | 管道传递 | 子查询链接 |

#### Gremlin（Apache TinkerPop）

Gremlin 采用命令式遍历范式，像在图上"走路"：

```groovy
// 找到张三工作的公司
g.V().has('Person', 'name', '张三')
  .out('WORKS_AT')
  .values('name')

// 多跳查询：张三的同事的朋友
g.V().has('Person', 'name', '张三')
  .out('WORKS_AT')
  .in('WORKS_AT')
  .out('FRIEND_OF')
  .dedup()
  .values('name')

// 最短路径
g.V().has('name', '张三')
  .repeat(both().simplePath())
  .until(has('name', '李四'))
  .path()
  .limit(1)
```

**Cypher vs Gremlin 对比**：

| 维度 | Cypher | Gremlin |
|------|--------|---------|
| 范式 | 声明式（像 SQL） | 命令式（像编程） |
| 学习曲线 | 较低（SQL 基础就能上手） | 较高（需理解遍历概念） |
| 表达力 | 模式匹配强 | 灵活遍历强 |
| 适用数据库 | Neo4j（原生）、部分兼容 | JanusGraph、Neptune 等 |
| 标准化 | GQL 标准基于 Cypher | Apache TinkerPop 标准 |

#### SPARQL（RDF 查询）

```sparql
# 找到所有在深圳的公司的员工
SELECT ?person ?company
WHERE {
  ?person ex:works_at ?company .
  ?company ex:located_in ex:Shenzhen .
}

# 联邦查询：跨多个 RDF 端点查询
SELECT ?name ?abstract
WHERE {
  SERVICE <http://dbpedia.org/sparql> {
    ?entity rdfs:label ?name .
    ?entity dbo:abstract ?abstract .
    FILTER(lang(?name) = "zh")
  }
}
```

#### GQL 标准化趋势

GQL（Graph Query Language）是 ISO 正在制定的图查询语言国际标准（ISO/IEC 39075），主要基于 Cypher 的语法风格。预计将成为图数据库领域的"SQL"，统一目前碎片化的查询语言生态。

### 2.3 图数据库存储引擎

**原生图存储 vs 非原生图存储**：

```
原生图存储 (Native Graph Storage):
┌─────────────────────────────────────────────┐
│  节点存储文件：固定大小记录                    │
│  [Node ID | Label | First Rel | First Prop] │
│                                              │
│  关系存储文件：双向链表                        │
│  [Rel ID | Type | Start | End | Next...]    │
│                                              │
│  特点：                                       │
│  • 邻接关系通过指针直接链接                    │
│  • 遍历时间与图大小无关（Index-Free Adjacency） │
│  • Neo4j 采用此方案                           │
└─────────────────────────────────────────────┘

非原生图存储 (Non-Native):
┌─────────────────────────────────────────────┐
│  底层用关系数据库或文档数据库存储               │
│  通过 JOIN 或索引模拟图遍历                    │
│                                              │
│  特点：                                       │
│  • 利用成熟的存储引擎                          │
│  • 遍历性能随图增大而下降                      │
│  • JanusGraph (Cassandra/HBase) 采用此方案    │
└─────────────────────────────────────────────┘
```

**Index-Free Adjacency（免索引邻接）** 是原生图存储的核心优势：每个节点直接持有指向其相邻节点的物理指针，遍历一条边的时间是 O(1)，不需要查索引。这意味着：
- **深度遍历性能恒定**：从一个节点出发走 10 跳，时间只与跳数有关，与图的总规模无关
- **对比关系数据库**：每一跳都是一次 JOIN，随着数据量增长 JOIN 越来越慢

### 2.4 大规模图的分布式处理

当图规模超过单机容量时，需要分布式策略：

**图分片（Graph Partitioning）**：

```
边切分 (Edge-Cut):
┌──────────┐     ┌──────────┐
│ Partition 1│     │ Partition 2│
│  A ── B    │────│  B ── C    │
│  A ── D    │     │  C ── E    │
│            │     │  D ── E    │
└──────────┘     └──────────┘
节点 B、D、E 可能被复制到多个分区

顶点切分 (Vertex-Cut):
每个边只存在于一个分区，顶点可能被切分
适合高度连接的图（如社交网络）
```

**核心挑战**：
1. **跨分片查询**：一次遍历可能需要跨多个分片，网络通信成为瓶颈
2. **负载均衡**：图的幂律分布（少数节点有大量连接）导致"热点"分片
3. **一致性**：分布式事务在图数据库中的实现复杂
4. **重分片**：数据增长后的重新分片代价高

**NebulaGraph 的存算分离架构**：
- Graph 层：负责查询解析、执行计划
- Storage 层：基于 Raft 一致性的分布式存储
- Meta 层：集群元数据管理
- 优势：存储和计算独立扩展，适合超大规模图

## 三、知识表示学习

### 3.1 为什么需要知识图谱嵌入？

知识图谱以符号形式存在（实体名 + 关系名），但很多下游任务需要**连续向量表示**：
- **链接预测**：预测缺失的三元组（如推断隐含关系）
- **实体对齐**：跨知识图谱匹配相同实体
- **推荐系统**：利用图谱关系增强推荐
- **知识补全**：发现知识图谱中缺失的事实

知识图谱嵌入（Knowledge Graph Embedding, KGE）的目标：将实体和关系映射到低维向量空间，使得图谱中的结构信息被保留。

### 3.2 平移模型（Translation-based Models）

#### TransE：开创性工作

**核心思想**：把关系看作从头实体到尾实体的"平移"操作。

```
数学表达：
  h + r ≈ t
  其中 h = 头实体嵌入, r = 关系嵌入, t = 尾实体嵌入

评分函数：
  f(h, r, t) = -||h + r - t||    （L1 或 L2 范数）
  
直觉理解：
  "张三" + "就职于" ≈ "腾讯"
  ───→     ───→       ───→
  向量空间中，实体沿着关系向量"平移"后应该接近目标实体
```

```
向量空间可视化：
       
        r (就职于)
    ───────────→
   h(张三)          t(腾讯)
    ●───────────→●
    
训练目标：
  • 正样本 (张三, 就职于, 腾讯)：h + r 尽量接近 t
  • 负样本 (张三, 就职于, 苹果)：h + r 远离 t'
  
损失函数（Margin-based）：
  L = Σ max(0, γ + f(h,r,t) - f(h',r,t'))
  γ 是 margin 超参数
```

**TransE 的局限**：
- **不能处理一对多/多对一/多对多关系**：如果"张三就职于腾讯"和"李四就职于腾讯"同时成立，那么 h₁ + r ≈ t 且 h₂ + r ≈ t，推出 h₁ ≈ h₂，即"张三≈李四"——显然不合理

#### TransR：关系空间投影

为了解决 TransE 的局限，TransR 引入了**关系特定的投影矩阵**：

```
核心思想：
  不同关系在不同的语义空间中操作
  
数学表达：
  h_r = M_r · h    （将实体投影到关系空间）
  t_r = M_r · t
  评分：f(h, r, t) = -||h_r + r - t_r||
  
  M_r 是关系 r 对应的投影矩阵（d×k 维）
  
直觉：
  "就职于"关系在"职业空间"中比较
  "毕业于"关系在"教育空间"中比较
  同一个实体在不同关系空间中有不同的表示
```

### 3.3 双线性模型（Bilinear Models）

#### DistMult

```
评分函数：
  f(h, r, t) = Σ_i h_i · r_i · t_i = <h, r, t>
  
即三个向量的逐元素乘积之和（三线性点积）

优点：简单高效
局限：评分函数关于 h 和 t 对称，无法建模非对称关系
  f(h, r, t) = f(t, r, h)  → "A 是 B 的父亲" = "B 是 A 的父亲"? 不对！
```

#### ComplEx

用**复数向量**解决 DistMult 的对称性问题：

```
实体和关系嵌入为复数向量：
  h = Re(h) + i·Im(h)
  r = Re(r) + i·Im(r)  
  t = Re(t) + i·Im(t)

评分函数：
  f(h, r, t) = Re(<h, r, conj(t)>)
  
其中 conj(t) 是 t 的共轭复数

关键性质：
  f(h, r, t) ≠ f(t, r, h)  → 可以建模非对称关系！
  当虚部为 0 时退化为 DistMult → 也能处理对称关系
```

### 3.4 旋转模型

#### RotatE

**核心思想**：关系建模为复数空间中的旋转操作。

```
数学表达：
  t = h ∘ r    （∘ 是逐元素复数乘法）
  |r_i| = 1   （关系向量的模为 1，即纯旋转）
  
评分函数：
  f(h, r, t) = -||h ∘ r - t||

直觉：
  关系是一个"旋转角度"
  对称关系 → 旋转 180°（r = -1）
  反对称关系 → 旋转非 180° 的角度
  逆关系 → 反方向旋转
  组合关系 → 旋转角度相加（r₁ ∘ r₂）

优势：
  能同时建模：对称、反对称、逆关系、组合关系
  这是 TransE 和 DistMult 都做不到的
```

### 3.5 各模型能力对比

```
┌──────────────┬────────┬─────────┬──────────┬──────────┐
│    模型       │ 对称关系 │ 反对称   │ 逆关系   │ 组合关系  │
├──────────────┼────────┼─────────┼──────────┼──────────┤
│ TransE       │   ✗    │   ✓     │   ✓      │   ✓     │
│ TransR       │   ✗    │   ✓     │   ✓      │   ✗     │
│ DistMult     │   ✓    │   ✗     │   ✗      │   ✗     │
│ ComplEx      │   ✓    │   ✓     │   ✓      │   ✗     │
│ RotatE       │   ✓    │   ✓     │   ✓      │   ✓     │
└──────────────┴────────┴─────────┴──────────┴──────────┘

对称关系示例：(A, 配偶, B) ↔ (B, 配偶, A)
反对称关系示例：(A, 父亲, B) 但 !(B, 父亲, A)
逆关系示例：(A, 父亲, B) ↔ (B, 孩子, A)
组合关系示例：(A, 父亲, B) ∧ (B, 父亲, C) → (A, 祖父, C)
```

### 3.6 嵌入质量评估

常用评估指标（以链接预测为例）：

```
任务：给定 (h, r, ?)，预测尾实体

评估流程：
1. 对所有候选实体打分
2. 按分数排序
3. 查看正确答案的排名

指标：
• Mean Rank (MR)：正确答案的平均排名（越小越好）
• Mean Reciprocal Rank (MRR)：排名倒数的平均值（越大越好）
  MRR = (1/N) Σ 1/rank_i
• Hits@K：正确答案排在前 K 位的比例（越大越好）
  Hits@1：精确预测能力
  Hits@10：前 10 候选的覆盖能力

过滤设置 (Filtered)：
  评估时排除其他正确答案，避免"假阴性"
  例如 (张三, 就职于, ?) 的答案可能有多个公司
```

> 💡 **面试考点**：TransE 为什么不能处理一对多关系？RotatE 如何同时建模对称和反对称关系？

## 四、知识图谱构建

知识图谱构建是将非结构化/半结构化数据转化为结构化三元组的过程，是整个知识图谱系统中最具挑战性的环节。

### 4.1 信息抽取流水线

```
┌──────────────────────────────────────────────────────┐
│                 知识图谱构建流水线                      │
│                                                       │
│  原始文本 → NER → 关系抽取 → 事件抽取 → 知识融合       │
│     ↓        ↓       ↓          ↓         ↓          │
│  "张三在腾讯  Person  works_at   入职事件   去重/       │
│   担任工程师"  Org     role_is    参与者    对齐        │
│              Location                                 │
│                                                       │
│                        ↓                              │
│                    知识图谱                             │
└──────────────────────────────────────────────────────┘
```

### 4.2 命名实体识别 (NER)

NER 是知识图谱构建的第一步——从文本中识别出实体及其类型。

**方法演进**：

```
传统方法：
  规则/词典 → HMM → CRF → BiLSTM-CRF
  
深度学习方法：
  BiLSTM-CRF → BERT-CRF → Span-based → GPT/LLM
  
  BiLSTM-CRF：
    输入 → BiLSTM（特征提取）→ CRF（序列解码）
    CRF 层确保标签序列合法（如 B-PER 后面只能接 I-PER 或 O）
  
  BERT-CRF：
    输入 → BERT（上下文编码）→ CRF（序列解码）
    BERT 的上下文表示大幅提升了 NER 效果
  
  Span-based：
    枚举所有可能的 span，对每个 span 分类
    避免了序列标注的标签依赖问题
    适合嵌套实体识别

LLM 方法（当前趋势）：
  直接用 Prompt 让 LLM 抽取实体
  优势：零样本/少样本，泛化能力强
  劣势：速度慢、成本高、输出不稳定
```

**LLM 做 NER 的 Prompt 设计**：

```
请从以下文本中识别所有命名实体，按 JSON 格式输出。
实体类型包括：Person(人物)、Organization(组织)、Location(地点)、
Date(日期)、Product(产品)。

文本：张三于2022年加入腾讯深圳总部，负责微信支付业务。

期望输出：
{
  "entities": [
    {"text": "张三", "type": "Person", "start": 0, "end": 2},
    {"text": "2022年", "type": "Date", "start": 3, "end": 8},
    {"text": "腾讯", "type": "Organization", "start": 10, "end": 12},
    {"text": "深圳", "type": "Location", "start": 12, "end": 14},
    {"text": "微信支付", "type": "Product", "start": 19, "end": 23}
  ]
}
```

### 4.3 关系抽取 (Relation Extraction)

关系抽取是从文本中识别实体对之间的语义关系。

**方法分类**：

```
1. 监督式关系抽取：
   • 管道式：先 NER，再对实体对分类
     输入：[张三] 就职于 [腾讯]
     分类器 → works_at
   • 联合抽取：同时识别实体和关系
     避免了误差传播
   
2. 远程监督 (Distant Supervision)：
   核心假设：如果知识图谱中已有 (张三, works_at, 腾讯)，
   那么同时包含"张三"和"腾讯"的句子大概率表达了 works_at 关系
   
   优势：自动生成大量训练数据
   劣势：假设过强，噪声大（"张三去腾讯面试被拒了"也会被标为 works_at）
   降噪方法：多实例学习 (Multi-Instance Learning)、注意力选择

3. 零样本/少样本关系抽取：
   用自然语言描述关系，让模型做文本蕴含判断
   "张三就职于腾讯" 蕴含 "一个人在一个公司工作" → works_at
```

**LLM 做关系抽取**：

```
从以下文本中抽取实体之间的关系，输出为三元组列表。
关系类型包括：works_at, graduated_from, located_in, founded_by, 
spouse_of, parent_of

文本：马化腾于1998年创立了腾讯公司，总部位于深圳南山区。
腾讯旗下的微信已成为国民级应用。

期望输出：
[
  {"head": "马化腾", "relation": "founded_by", "tail": "腾讯"},
  {"head": "腾讯", "relation": "located_in", "tail": "深圳南山区"},
  {"head": "微信", "relation": "subsidiary_of", "tail": "腾讯"}
]
```

### 4.4 事件抽取

事件抽取比关系抽取更复杂，需要识别：
- **事件类型**：入职、离职、收购、合并...
- **触发词**：表示事件发生的词语
- **论元角色**：事件参与者及其角色

```
示例：
文本："2022年3月，腾讯以27亿美元收购了黑鲨科技。"

抽取结果：
{
  "event_type": "收购",
  "trigger": "收购",
  "arguments": [
    {"role": "收购方", "entity": "腾讯"},
    {"role": "被收购方", "entity": "黑鲨科技"},
    {"role": "金额", "entity": "27亿美元"},
    {"role": "时间", "entity": "2022年3月"}
  ]
}
```

### 4.5 知识融合

当从多个来源抽取知识后，需要融合成统一的知识图谱：

**实体消歧（Entity Disambiguation）**：
```
"苹果" → Apple Inc.（科技公司）还是 🍎（水果）？
方法：
  • 上下文消歧：看周围词语（"手机"→公司，"维生素"→水果）
  • 实体链接：将文本中的实体链接到知识库中的标准实体
  • 向量相似度：实体嵌入 + 上下文嵌入的匹配
```

**实体对齐（Entity Alignment）**：
```
跨知识图谱匹配相同实体：
  Wikidata: Q312 → Apple Inc.
  DBpedia: dbr:Apple_Inc.
  
方法：
  • 基于属性匹配：名称、类型、关键属性的相似度
  • 基于嵌入对齐：MTransE、GCN-Align 等
  • 基于 LLM：让大模型判断两个实体描述是否指同一实体
```

**知识冲突解决**：
```
来源1：腾讯成立于1998年
来源2：腾讯成立于1999年

解决策略：
  • 投票法：多数来源一致的值优先
  • 可信度加权：根据来源可靠性加权
  • 时效性优先：最新来源的信息优先
  • 人工审核：高冲突的交给人工
```

### 4.6 LLM 辅助知识图谱构建

这是当前最热门的方向——用大模型作为知识抽取的核心引擎：

**端到端三元组抽取 Prompt**：

```

要求：
1. 识别所有实体（人物、组织、地点、产品、时间等）

4. 关系要具体明确，避免使用"相关"这种模糊关系

文本：
"""
{text}
"""

输出示例格式：
{
  "triples": [
    {
      "head": {"name": "...", "type": "..."},
      "relation": "...",
      "tail": {"name": "...", "type": "..."},

```

## 五、图神经网络 (GNN)

### 5.1 核心思想：消息传递与邻居聚合

GNN 的核心思想一句话概括：**一个节点的特征，由它的邻居决定**。

这背后的直觉是"物以类聚"——你是什么样的人，看你周围是什么样的人就知道了。

```
消息传递框架 (Message Passing Neural Network, MPNN)：

第 k 层的节点 v 的表示更新：

  h_v^(k) = UPDATE(h_v^(k-1), AGGREGATE({h_u^(k-1) : u ∈ N(v)}))

分三步：
  1. MESSAGE：每个邻居节点 u 生成一条"消息"
  2. AGGREGATE：聚合所有邻居的消息（求和、平均、最大值等）
  3. UPDATE：用聚合结果更新当前节点的表示

可视化：
                聚合
    邻居1 ──→ ┐
    邻居2 ──→ ├──→ 更新 ──→ 节点v的新表示
    邻居3 ──→ ┘
    
多层堆叠 = 更大感受野：
  1 层 GNN：看到 1 跳邻居
  2 层 GNN：看到 2 跳邻居（邻居的邻居）
  K 层 GNN：看到 K 跳邻居
```

### 5.2 关键架构演进

#### GCN（Graph Convolutional Network）

```
GCN 更新公式：
  H^(l+1) = σ(D̃^(-1/2) Ã D̃^(-1/2) H^(l) W^(l))

其中：
  Ã = A + I  （邻接矩阵 + 自连接）
  D̃ 是 Ã 的度矩阵
  W^(l) 是可学习的权重矩阵
  σ 是激活函数

直觉：
  D̃^(-1/2) Ã D̃^(-1/2) 做的是"归一化的邻居平均"
  自连接 (I) 确保节点也考虑自己的信息
  
局限：
  所有邻居权重相同（由度决定），无法学习"谁更重要"
```

#### GAT（Graph Attention Network）

GAT 引入注意力机制，让模型学习不同邻居的重要性权重：

```
GAT 更新公式：

  1. 计算注意力系数：
     e_ij = LeakyReLU(a^T [W·h_i || W·h_j])
     α_ij = softmax_j(e_ij) = exp(e_ij) / Σ_k exp(e_ik)
  
  2. 加权聚合：
     h_i' = σ(Σ_{j∈N(i)} α_ij · W·h_j)
  
  3. 多头注意力：
     h_i' = ||_{k=1}^{K} σ(Σ_{j∈N(i)} α_ij^k · W^k·h_j)

直觉：
  不是所有朋友都同样重要
  GAT 会自动学习：在判断张三的职业时，
  "同事"关系比"高中同学"关系更有信息量
```

#### GraphSAGE：归纳式学习

前面的 GCN/GAT 是**直推式（Transductive）**的——只能处理训练时见过的节点。GraphSAGE 实现了**归纳式（Inductive）**学习：

```
GraphSAGE 流程：
  1. 采样：对每个节点的邻居进行采样（固定数量）
  2. 聚合：用学习到的聚合函数聚合邻居特征
  3. 组合：将聚合结果与自身特征组合

  h_v^(k) = σ(W · CONCAT(h_v^(k-1), AGG({h_u^(k-1): u ∈ Sample(N(v))})))

聚合函数选择：
  • Mean Aggregator：简单平均
  • LSTM Aggregator：将邻居排序后用 LSTM 处理
  • Pool Aggregator：先变换后取最大值

关键优势：
  学到的是"聚合策略"而非"每个节点的嵌入"
  → 新加入的节点不需要重新训练
  → 适合动态图、大规模图
```

#### GIN（Graph Isomorphism Network）

```
GIN 的理论贡献：
  证明了 GNN 的表达力上限等价于 WL 图同构检验
  
  大多数 GNN（GCN、GraphSAGE）的聚合函数不够"强"
  GIN 使用 sum 聚合 + MLP 更新，达到了理论上限

  h_v^(k) = MLP((1 + ε) · h_v^(k-1) + Σ_{u∈N(v)} h_u^(k-1))

意义：
  如果两个图的 GIN 嵌入不同，那么它们一定是不同构的
  这为 GNN 的理论分析奠定了基础
```

### 5.3 图级别任务 vs 节点级别 vs 边级别

```
┌──────────┬──────────────────────────────────────────────┐
│ 任务级别  │                 说明与示例                     │
├──────────┼──────────────────────────────────────────────┤
│ 节点级别  │ 预测每个节点的标签                             │
│          │ • 社交网络中预测用户兴趣                        │
│          │ • 论文引用网络中预测论文类别                     │
│          │ • 欺诈检测中识别可疑账号                        │
├──────────┼──────────────────────────────────────────────┤
│ 边级别   │ 预测两个节点之间是否存在边（链接预测）           │
│          │ • 知识图谱补全                                 │
│          │ • 推荐系统（用户-物品连接预测）                  │
│          │ • 社交网络中预测潜在好友                        │
├──────────┼──────────────────────────────────────────────┤
│ 图级别   │ 预测整个图的属性                               │
│          │ • 分子属性预测（毒性、溶解度）                   │
│          │ • 程序分析（代码图分类）                        │
│          │ 需要 Graph Readout（将节点嵌入池化为图嵌入）     │
└──────────┴──────────────────────────────────────────────┘
```

### 5.4 过平滑问题

**问题描述**：当 GNN 层数增加时，所有节点的表示趋于相同，区分度消失。

```
直觉理解：
  1 层：每个节点只看 1 跳邻居 → 保留局部信息
  2 层：看 2 跳邻居 → 更多上下文
  ...
  K 层：看 K 跳邻居 → 当 K 很大时，所有节点都"看到"了整个图
  → 所有节点的表示都变成了"图的全局平均" → 无法区分

数学解释：
  GCN 的传播类似于 Laplacian 平滑
  多次平滑后，信号趋于常数（低频分量）
  
解决方案：
  1. 残差连接：h_v^(k) = h_v^(k) + h_v^(k-1)
  2. JumpingKnowledge：拼接所有层的输出
  3. DropEdge：训练时随机删边，减缓过平滑
  4. PairNorm：归一化保持节点间的距离
  5. 实践建议：大多数场景 2-3 层 GNN 就够了
```

### 5.5 异构图神经网络

现实中的知识图谱是**异构图**——有多种类型的节点和边。

```
异构知识图谱示例：
  节点类型：Person, Company, University, City
  边类型：works_at, graduated_from, located_in, founded_by

处理异构图的方法：

1. 关系感知 GNN (R-GCN)：
   每种关系类型有自己的权重矩阵
   h_v^(l+1) = σ(Σ_{r∈R} Σ_{u∈N_r(v)} (1/c_{v,r}) W_r^(l) h_u^(l) + W_0^(l) h_v^(l))
   
   问题：关系类型多时参数爆炸
   解决：基分解 (Basis Decomposition) 或块对角分解

2. 异构图 Transformer (HGT)：
   用异构注意力机制处理不同类型的节点和边
   注意力权重取决于：源节点类型、边类型、目标节点类型

3. 元路径方法 (Metapath-based)：
   定义有意义的元路径：Person → works_at → Company → located_in → City
   沿着元路径聚合信息
   如 HAN (Heterogeneous Attention Network)
```

## 六、KG + LLM 融合

这是当前知识图谱领域最热门、最有实际价值的方向。KG 和 LLM 的融合是双向的：KG 增强 LLM，LLM 也增强 KG。

### 6.1 KG 增强 LLM

#### Graph RAG：用知识图谱结构化信息增强生成

传统 RAG 的局限：
- 检索的是独立的文本片段，缺乏**全局结构**
- 对"总结性问题"（如"这个领域有哪些主要趋势？"）表现差
- 相关信息可能分散在多个文档中，难以关联

Graph RAG 的解决思路：

```
┌──────────────────────────────────────────────────────┐
│                  Graph RAG 架构                       │
│                                                       │
│  ┌─────────────────────────────────────────────────┐ │
│  │              离线索引阶段                         │ │
│  │                                                  │ │
│  │  原始文档 → 文本分块 → LLM实体/关系抽取          │ │
│  │                  ↓                               │ │
│  │          构建知识图谱                             │ │
│  │                  ↓                               │ │
│  │     社区检测(Leiden算法) → 生成社区摘要            │ │
│  │                                                  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  ┌─────────────────────────────────────────────────┐ │
│  │              在线查询阶段                         │ │
│  │                                                  │ │
│  │  用户查询 → 查询分析 → 选择检索策略              │ │
│  │                                                  │ │
│  │  Local Search:                                   │ │
│  │    相关实体 → 邻居遍历 → 关联文本 → 生成回答     │ │
│  │                                                  │ │
│  │  Global Search:                                  │ │
│  │    社区摘要 → Map(各社区回答) → Reduce(综合)     │ │
│  │                                                  │ │
│  └─────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
```

**Microsoft GraphRAG 核心流程详解**：

**Step 1：文档 → 知识图谱**
```
输入：一批文档（PDF、文本等）
处理：
  1. 文本分块（通常 300-600 tokens）
  2. 对每个块用 LLM 抽取实体和关系
  3. 合并所有块的抽取结果，去重、对齐
  4. 得到完整的知识图谱
```

**Step 2：社区检测**
```
用 Leiden 算法对知识图谱做社区检测
将图分成多个"社区"（紧密连接的子图）

为什么？
  社区 = 主题聚类
  一个社区通常对应一个话题或概念群
  不同层级的社区对应不同粒度的主题
```

**Step 3：社区摘要生成**
```
对每个社区，用 LLM 生成摘要
摘要描述了该社区的主要实体、关系和核心主题

这些摘要就是 Global Search 的基础
```

**Step 4：查询时的两种策略**

```
Local Search（局部搜索）：
  适合：具体的、有明确实体的问题
  如："张三在哪个公司工作？"
  流程：找到实体 → 遍历邻居 → 收集相关文本 → 生成回答

Global Search（全局搜索）：
  适合：总结性、概览性的问题
  如："这个领域有哪些主要趋势？"
  流程：
    Map 阶段：对每个社区摘要生成部分回答
    Reduce 阶段：综合所有部分回答生成最终答案
```

**Graph RAG vs 传统 RAG 对比**：

```
┌──────────────┬─────────────────┬──────────────────────┐
│    维度       │   传统 RAG       │    Graph RAG         │
├──────────────┼─────────────────┼──────────────────────┤
│ 索引结构     │ 向量索引         │ 知识图谱 + 社区结构   │
│ 检索方式     │ 语义相似度       │ 图遍历 + 社区匹配     │
│ 上下文质量   │ 独立文本片段     │ 结构化关联信息        │
│ 全局问题     │ 差               │ 优秀（社区摘要）      │
│ 细节问题     │ 好              │ 好（局部搜索）         │
│ 索引成本     │ 低（一次嵌入）   │ 高（LLM 抽取）        │
│ 维护成本     │ 低              │ 高（图谱更新）         │
│ 适用场景     │ 通用 QA         │ 复杂知识库、分析型问题  │
└──────────────┴─────────────────┴──────────────────────┘
```

#### KG-Grounded Generation：基于图谱的事实验证

```
流程：
  1. LLM 生成初始回答
  2. 从回答中抽取事实声明（claims）
  3. 每个声明去知识图谱中验证
  4. 标记已验证/未验证/矛盾的声明
  5. 基于验证结果修正回答

示例：
  LLM 回答："腾讯由马化腾于1999年在北京创立。"
  
  KG 验证：
  ✓ (腾讯, founded_by, 马化腾) → 存在
  ✗ (腾讯, founded_year, 1999) → KG 中是 1998 → 矛盾！
  ✗ (腾讯, founded_location, 北京) → KG 中是 深圳 → 矛盾！
  
  修正后："腾讯由马化腾于1998年在深圳创立。"
```

#### 结构化推理：利用图谱辅助多跳推理

```
问题："张三的老板的母校在哪个城市？"

传统 LLM：可能直接胡编一个答案

KG 辅助推理：
  Step 1: 图谱查询 (张三)-[boss]->(李四) → 找到老板
  Step 2: 图谱查询 (李四)-[graduated_from]->(清华大学) → 找到母校  
  Step 3: 图谱查询 (清华大学)-[located_in]->(北京) → 找到城市
  
  将推理路径作为上下文给 LLM：
  "根据知识图谱：张三的老板是李四，李四毕业于清华大学，
   清华大学位于北京。因此答案是北京。"
```

### 6.2 LLM 增强 KG

#### 用 LLM 做知识抽取和图谱构建

这在第四章已经详细讨论。核心优势：
- **零样本能力**：不需要标注数据
- **通用性强**：一个 Prompt 适配多个领域
- **理解上下文**：能处理复杂的语言表达

#### LLM 辅助本体设计

```
Prompt 示例：

你是一位本体工程师。请为"智能客服"领域设计一个知识图谱本体。

要求：
1. 列出核心实体类型（至少 8 种）
2. 定义实体之间的关系（至少 15 种）
3. 为每种实体类型列出关键属性
4. 说明类之间的继承关系
5. 考虑可扩展性

LLM 输出：
{
  "entity_types": [
    {"name": "Product", "parent": "Entity", "attributes": ["name", "category", "price", "description"]},
    {"name": "Issue", "parent": "Entity", "attributes": ["type", "severity", "description"]},
    {"name": "Solution", "parent": "Entity", "attributes": ["steps", "effectiveness", "applicable_products"]},
    ...
  ],
  "relations": [
    {"name": "has_issue", "domain": "Product", "range": "Issue"},
    {"name": "resolved_by", "domain": "Issue", "range": "Solution"},
    ...
  ]
}
```

#### Text-to-Cypher / Text-to-SPARQL

让用户用自然语言查询知识图谱，LLM 翻译成图查询语言：

```
用户："找出所有在深圳的互联网公司，以及它们的创始人"

LLM 翻译为 Cypher：
MATCH (c:Company)-[:LOCATED_IN]->(city:City {name: "深圳"})
WHERE c.industry = "互联网"
OPTIONAL MATCH (p:Person)-[:FOUNDED]->(c)
RETURN c.name AS company, p.name AS founder

关键挑战：
  1. 需要给 LLM 提供图谱的 Schema 信息
  2. 实体名称需要标准化（"深圳" vs "深圳市"）
  3. 查询正确性验证
  4. 处理歧义和模糊查询

最佳实践：
  • 在 Prompt 中提供完整的 Schema（节点标签、关系类型、属性名）
  • 提供 Few-shot 示例
  • 生成查询后先验证语法，再执行
  • 对执行结果做二次验证
```

### 6.3 知识图谱在 Agent 系统中的角色

知识图谱作为 Agent 的**结构化长期记忆**：

```
┌─────────────────────────────────────────────────┐
│                Agent 记忆架构                     │
│                                                   │
│  ┌──────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ 工作记忆  │  │ 向量记忆      │  │ 图谱记忆   │ │
│  │ (短期)    │  │ (语义检索)    │  │ (结构化)   │ │
│  │          │  │              │  │            │ │
│  │ 当前对话  │  │ 历史对话     │  │ 实体关系   │ │
│  │ 上下文    │  │ 文档片段     │  │ 因果链     │ │
│  │ 中间状态  │  │ 经验总结     │  │ 时间线     │ │
│  └──────────┘  └──────────────┘  └────────────┘ │
│       ↑               ↑               ↑          │
│       └───────────────┴───────────────┘          │
│                    Agent Core                     │
│              (LLM + 规划 + 执行)                  │
└─────────────────────────────────────────────────┘

知识图谱记忆的独特价值：
  1. 关系推理：沿着图遍历发现隐含关系
  2. 时间推理：事件的先后顺序、因果关系
  3. 一致性保证：避免自相矛盾的记忆
  4. 可解释性：推理路径可追溯
```

### 6.4 知识图谱与向量检索的融合检索

在实际系统中，最佳实践是将知识图谱检索和向量检索结合：

```
混合检索策略：

用户查询 → 查询分析
                ↓
    ┌──────────┴──────────┐
    ↓                      ↓
 向量检索                图谱检索
 (语义相似)             (结构遍历)
    ↓                      ↓
 文本片段 Top-K         关联实体/关系
    ↓                      ↓
    └──────────┬──────────┘
               ↓
          结果融合与重排序
               ↓
          LLM 生成回答

融合策略：
  1. 互补检索：向量找相关文本，图谱找结构化事实
  2. 图谱引导向量检索：先在图谱中找到相关实体，再用实体名做向量检索
  3. 向量引导图谱遍历：先向量检索找到起始实体，再在图谱中扩展
```

## 七、技术选型决策指南

### 7.1 图数据库选型

```
场景判断流程：

Q: 数据规模？
├── < 1亿节点 → Neo4j（社区版免费，生态好）
├── 1-100亿节点 → NebulaGraph / TigerGraph
└── > 100亿节点 → TigerGraph / 自研

Q: 团队技术栈？
├── AWS 生态 → Amazon Neptune
├── 有 Hadoop 集群 → JanusGraph
├── 国内团队 → NebulaGraph（中文文档、社区活跃）
└── 初创/小团队 → Neo4j（学习资源最多）

Q: 需要 RDF 支持？
├── 是（语义网、开放数据）→ Amazon Neptune / Apache Jena
└── 否（业务应用）→ Neo4j / NebulaGraph
```

### 7.2 知识图谱构建方法选型

```
Q: 有标注数据吗？
├── 有大量标注 → 监督学习方法（BERT-CRF 等）
├── 有少量标注 → Few-shot 微调 + LLM 辅助
└── 无标注 → 纯 LLM 方法

Q: 领域专业度？
├── 通用领域 → 直接 LLM
├── 专业领域（医学、法律）→ 领域微调 + 专家审核
└── 混合 → LLM 初抽取 + 专业规则后处理

Q: 数据规模？
├── 小规模（< 1万文档）→ LLM 端到端
├── 中规模（1万-100万）→ LLM 抽取 + 批处理优化
└── 大规模（> 100万）→ 传统 NLP 方法 + LLM 验证
```

### 7.3 KG + LLM 融合方案选型

```
Q: 核心需求是什么？
├── 减少幻觉 → KG-Grounded Generation
├── 复杂知识问答 → Graph RAG
├── 自动构建知识库 → LLM-driven KG Construction
├── 自然语言查图谱 → Text-to-Cypher
└── Agent 长期记忆 → KG as Memory Store

Q: 成本预算？
├── 高（企业级）→ 完整 Graph RAG 方案
├── 中 → 简化版 KG 增强 RAG
└── 低 → 小规模知识图谱 + 规则检索
```

## 附录：面试高频考点速查

### 基础概念类

| 考点 | 核心回答要点 |
|------|-------------|
| RDF vs 属性图 | RDF：W3C标准、严格三元组、URI标识、SPARQL查询；属性图：边可有属性、更灵活、Cypher查询、工程主流 |
| 知识图谱 vs 关系数据库 | KG 优势在多跳关系查询（O(K) vs O(N^K)）；RDBMS 优势在事务和精确查询 |
| 本体的作用 | 定义 Schema、类层次、关系约束、支持推理；类比于数据库的 DDL |

### 技术深度类

| 考点 | 核心回答要点 |
|------|-------------|
| TransE 原理及局限 | h+r≈t，简单高效；局限：不能建模一对多/多对一关系（会导致实体嵌入退化） |
| RotatE 为什么更好 | 关系=复数空间旋转；对称(180°)、反对称(非180°)、逆(反向)、组合(角度加)全覆盖 |
| GNN 过平滑 | 层数增加→所有节点表示趋同→类似 Laplacian 平滑；解决：残差、JumpingKnowledge、DropEdge |
| R-GCN 处理异构图 | 每种关系一个权重矩阵；参数爆炸用基分解缓解 |

### 应用实践类

| 考点 | 核心回答要点 |
|------|-------------|
| Graph RAG vs 传统 RAG | Graph RAG 优势：结构化上下文、全局问题能力（社区摘要）；劣势：索引成本高 |
| LLM 构建 KG 的挑战 | Schema 定义要清晰、实体标准化、关系映射、质量验证、成本控制 |
| Text-to-Cypher | 需提供 Schema、Few-shot 示例、语法验证、结果验证；处理歧义是关键 |
| KG 在 Agent 中的角色 | 结构化长期记忆、关系推理、时间推理、一致性保证、可解释性 |

### 选型决策类

| 考点 | 核心回答要点 |
|------|-------------|
| Neo4j vs NebulaGraph | Neo4j：生态好、学习成本低、单机优秀；NebulaGraph：分布式强、大规模、国产 |
| 何时用 KG 而非 RDBMS | 关系复杂（>3跳频繁）、需要路径分析、知识推理、数据模式灵活多变 |
| NER 方法选择 | 标注多→BERT-CRF；标注少→Few-shot+LLM；无标注→纯 LLM；速度要求高→小模型 |
